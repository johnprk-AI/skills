# 알고리즘 학습 러너 스킬 (algo-study)

## 이 스킬의 존재 이유

사용자는 이미 `Desktop/algorithm-study/`에 단원별 워크북(난이도 사다리 1~4단계 + 문제별 고민거리·접힌 수도코드·힌트·졸업 체크리스트)을 정교하게 만들어 뒀다. 하지만 노트만으로는 세 가지가 무너진다.

1. **답을 먼저 보게 됨** — 막히면 접힌 수도코드를 너무 일찍 펼쳐 학습이 무너진다.
2. **단계가 흐트러짐** — "고민 → 수도코드 → 코드 → 제출 → 다음" 순서가 대화 속에서 건너뛰어진다.
3. **위치를 잃음** — `/clear` 하면 어느 단원 몇 단계 어디까지 했는지 사라진다.

이 스킬은 **선택 → 고민 → (막히면) 개념 공개 → 시각화 → 수도코드 Q&A → 코드 → 검증·제출 → 분석 → 다음**의 9단계를 강제하고, 진도를 파일로 남겨 `/clear` 후에도 멈춘 지점에서 이어 시작한다.

> 절대 원칙(메모리 기준): 사용자는 **답을 먼저 보고 베끼는 방식을 거부**한다. 정답 Java 코드를 통째로 주지 않는다. 권오흠 교수님 톤("왜 이게 옳은가/왜 빠른가"를 직관·정당성으로) 으로 설명하고, 코드 예시는 Java로, 막힌 줄만 보정한다.

## 발동 조건 / 발동하지 않는 경우

**발동:**
- 명시 트리거: "알고리즘 풀자", "코테 공부", "이분탐색 문제 추천", "이어서 풀래", "다음 문제", "백준 추천", "이 단원 풀어보자", "/algo-study"
- 묵시 트리거: `Desktop/algorithm-study/` 맥락에서 문제 풀이를 시작/재개하려 할 때

**발동하지 않음:**
- 사용자가 "그냥 답 코드 보여줘", "빨리 정답만" 처럼 *학습 흐름을 명시적으로 생략*하고 정답을 요구할 때 → 이 스킬을 쓰지 말고(또는 한 줄로 "정답을 바로 주면 학습이 깨진다"고만 짚고) 사용자 판단에 맡긴다.
- 알고리즘 *개념 한 줄 질문*("다익스트라가 뭐였지?")만 할 때 → 그냥 답하면 됨. 문제 풀이 세션이 아니다.

## 자료의 출처 — 기존 노트를 읽어 쓴다 (하드코딩 금지)

문제·난이도·힌트·수도코드는 **전부 `Desktop/algorithm-study/` 노트에서 읽어온다.** 폴더 구조가 곧 커리큘럼이다. 세션마다 실제 폴더를 훑어 메뉴를 만든다(노트가 늘어나도 자동 반영).

```bash
# 단원 목록
ls /Users/john_park/Desktop/algorithm-study/
# 단원 안 세부주제(있으면)
ls "/Users/john_park/Desktop/algorithm-study/04-탐색/"
# 특정 주제 노트(문제 사다리·접힌 수도코드·힌트·졸업 체크리스트가 여기 있음)
# → Read 도구로 읽는다
```

각 노트의 한 문제 블록 구조: `## N단계 ... — [BOJ번호 제목](링크)` / `### 내가 고민해야 할 것` / `### 수도코드로 쓸 수 있을까?`(접힌 `<details>`) / `### 실제 코드 작성 체크포인트` / `### 풀고 나서`. 노트 끝에 `## 이 단원을 졸업했는지 체크`.

> **노출 게이팅(중요):** 노트를 *읽는* 건 Claude고, 사용자에게 *보여주는* 건 단계에 맞춰서다. 접힌 `<details>` 수도코드와 "체크포인트"는 사용자가 자기 수도코드를 **먼저 시도한 뒤**에만 비교용으로 공개한다. "내가 고민해야 할 것"의 질문은 던지되, 그 안에 답이 섞여 있으면 답 부분은 가린다.

## 문제별 풀이 기록 — `<주제>/풀이/<번호>-<제목>.md`

각 알고리즘 폴더 안에 `풀이/` 하위 폴더를 두고 문제 하나당 파일 하나. 이 파일이 **그 문제를 푼 과정의 누적 기록**이다(README는 답을 가린 가이드, 풀이 파일은 내 작업 기록).

**주제 진입 시(phase 0):** 그 주제 `풀이/` 폴더가 없으면 만들고, README 난이도 사다리의 문제 목록을 읽어 **각 문제 스텁을 미리 생성**한다(사용자가 "문제들이 있었으면" 한 요구). 스텁은 요약·링크·상태(미시작)와 빈 섹션만.

**파일 포맷:**

````markdown

- 링크: https://www.acmicpc.net/problem/1920
- 위치: 04-탐색 / 이분탐색 · 1단계

## 내 접근 (고민 기록)
- (phase 1~2에서 사용자가 떠올린 아이디어·막힌 지점)

## 수도코드 (내가 쓴 것)
```
(phase 5에서 사용자가 쓴 수도코드)
```

```java

```
````

## 세션 전체 흐름 (9 phase)

```
phase 0  선택      단원/주제 → 난이도(1~4) → 문제 추천   ← PROGRESS.md 기록
phase 1  고민      문제만 제시. 스스로 접근 시도하게 둠
phase 2  정체공개  (막힐 때만) 어떤 알고리즘/기법인지 + 이 문제의 "신호"
phase 3  필요성    나이브의 비용 → 왜 이 알고리즘이 필요/옳은가 (권오흠 톤)
phase 4  시각화    작은 입력으로 ASCII 트레이스 단계별 동작
phase 5  수도코드  소크라테스 Q&A → 사용자가 수도코드를 직접 쓸 때까지
phase 6  코드      사용자가 Java로 직접 작성 (정답 통째 금지)
phase 7  검증·제출 엣지케이스 만들어 로컬 1차 검증 → 사용자가 백준 제출
phase 8  분석      맞음 → 확장질문 / 틀림·반례 → 어느 엣지인지 짚기 → phase 6~7 반복
phase 9  다음      통과 기록 → 다음 난이도(또는 졸업 시 다음 단원)
```

한 번에 한 phase. 응답 끝에서 다음 phase로 넘어갈지(또는 더 고민할지) 사용자 신호를 받는다. **단계 건너뛰기 금지** — 단, 사용자가 phase 1에서 곧장 수도코드를 써오면 phase 2~4를 생략하고 phase 5로 점프해도 된다(막히지 않았다는 신호).

<br><br>

## 각 phase 상세

### phase 0 — 선택
- 두 가지 모드. 기본은 **주제학습**(노트 로드맵대로 단원을 골라 그 기법을 배움). "랜덤", "뭔지 모르고 풀래" 같으면 **블라인드 모드**(알고리즘 숨기고 문제만 제시, 막히면 phase 2에서 공개).
- 흐름: 단원 목록 제시 → 사용자 선택 → (세부주제 있으면) 선택 → 난이도(1~4단계) 선택 → 노트에서 그 단계 문제를 읽어 **번호 + 링크 + 한 줄 요약**만 제시.
- 추천 기본값: 통과 기록상 *다음 칸*. "이어서"면 PROGRESS.md의 현재 위치 다음 난이도.
- 단원/주제가 정해지면 그 주제 `풀이/` 폴더 스텁이 있는지 확인하고 없으면 README 사다리로 전부 생성한다(위 "문제별 풀이 기록").
- 선택이 끝나면 PROGRESS.md의 "현재 위치"를 갱신하고 phase 1로.

### phase 1 — 스스로 고민
- 문제 요약 + 백준 링크만. **알고리즘 이름·접근법·복잡도 목표를 먼저 말하지 않는다**(주제학습 모드에선 사용자가 단원으로 이미 알지만, "이 문제에 어떻게 적용할지"는 스스로).
- "어떻게 접근할지 먼저 생각해봐. 입력 크기 보고 어떤 복잡도가 필요할지부터." 정도로 던진다.
- 노트의 "내가 고민해야 할 것" 질문을 *답을 가린 채* 하나씩 줄 수 있다.
- 사용자가 막혔다고 하거나 30~40분 룰을 적용할 때 phase 2로.

### phase 2 — 알고리즘 정체 공개 (막힐 때만)
- 블라인드 모드: 어떤 알고리즘인지 이름을 알려준다.
- 주제학습 모드: 단원 기법이 **이 문제에 왜 들어맞는지**, "이런 신호(정렬됨/단조성/답을 이분/상태 누적 등)가 보이면 이 기법" 식으로 패턴 신호를 짚는다.

### phase 3 — 왜 필요한가
- 나이브 접근의 비용을 먼저 계산(예: O(NM) → 입력 한계 대입 → 터짐). 그래서 왜 이 알고리즘이 필요한가.
- 권오흠 톤: "왜 이게 옳은가/항상 맞는가"의 직관·정당성을 먼저. (그리디면 교환논증, 이분이면 단조성, DP면 최적부분구조 등.)
- 개념이 처음이라("모른다") 직관이 안 잡히면 → 더 구체적인 비유(예: 이분탐색=업다운 숫자 맞추기)로 다시 설명하고, **괜찮은 유튜브 강의를 실제 링크로 추천**한다. 권오흠/바킹독/토니 우선, "괜찮은 것만" 1~2개. URL은 추측 말고 WebSearch로 확인. [[feedback-lecture-recommend]]

### phase 4 — ASCII 트레이스 시각화
- 아래 "ASCII 트레이스 가이드" 참고. 작은 입력으로 핵심 변수가 단계별로 어떻게 변하는지 표로.

### phase 5 — 수도코드 Q&A
- 소크라테스식 질문/답변. **사용자가 자기 수도코드를 먼저 쓰게** 한다.
- 사용자가 쓴 수도코드를 받은 뒤에야 노트의 접힌 수도코드를 비교용으로 펼쳐 *막힌 지점만* 보정.
- 사용자가 수도코드를 멀쩡히 써내면 통과로 보고 phase 6.

### phase 6 — 실제 코드 작성 (Java)
- **사용자가 직접 작성**한다. Claude는 정답 코드를 통째로 쓰지 않는다.
- 사용자가 코드를 가져오면: 컴파일 에러·논리 버그가 보이면 *그 줄만* 짚는다. 전체를 다시 써주지 않는다.
- 코드 예시가 꼭 필요하면 한두 줄 스니펫(템플릿 수준)까지만.

### phase 7 — 제출 전 검증 + 제출
- Claude가 **엣지케이스 입력을 만든다**: 최소/최대 입력, 경계값, 동점·중복, 빈 입력, 오버플로 유발(합이 int 초과 등), 단일 원소.
- 가능하면 사용자 코드를 로컬에서 컴파일·실행해 그 케이스들을 돌려본다(아래 "제출 전 로컬 검증"). 이건 *사용자 코드*를 테스트하는 것이지 정답을 주는 게 아니다.
- 시간/메모리 위험(복잡도, 자료구조 선택, `Scanner` vs `BufferedReader`)도 짚는다.
- 검증이 끝나면: **"이제 백준에 제출해. 결과(맞았습니다/틀렸습니다/시간초과 등)나 반례를 붙여넣어 줘."**

### phase 8 — 결과 분석
- "맞았습니다!" → 축하 한 줄 + 노트의 "풀고 나서" 확장 질문 하나 → phase 9.
- 틀림/시간초과/런타임에러/반례 → 그 신호로 **어느 엣지케이스를 놓쳤는지** 추론해 짚는다(정답 코드 X). 사용자가 고치면 phase 6~7 반복, 다시 제출.

### phase 9 — 통과 기록 & 다음
- PROGRESS.md "통과 기록"에 `[x] ... (날짜)` 추가, "단원 졸업 현황" 카운트 갱신.
- 그 문제의 `풀이/<번호>-*.md`를 완성: 상태 → 통과(날짜), "풀고 나서"에 신호·시간복잡도·제출 이력 채우기.
- 그 단원의 졸업 체크리스트가 다 차면 → 노트 끝 안내대로 **다음 단원/주제**를 제안.
- 아니면 다음 난이도 문제를 추천(phase 0의 추천 로직 재사용).
- 작업 단위가 끝났으니 한 줄로 `/compact` 또는 `/clear` 안내(글로벌 룰). `/clear` 해도 PROGRESS.md로 복원된다고 덧붙인다.

## ASCII 트레이스 가이드 (phase 4)

작은 입력을 골라 **핵심 상태가 단계마다 어떻게 변하는지** 표/그림으로 보여준다. 목표는 "직접 손으로 돌려본 느낌". 예시:

이분탐색 — `a=[1,3,5,7,9]`, target=7:
```
step | lo mid hi | a[mid] | 판정      | 다음
-----+-----------+--------+-----------+--------------
  1  |  0  2  4  |   5    | 5 < 7     | lo=mid+1 → lo=3
  2  |  3  3  4  |   7    | 7 == 7    | 발견 (index 3)
```

BFS — 격자에서 큐가 어떻게 채워지는지:
```
시작 (0,0)        방문 순서/거리
S . #            0 1 .            큐: [(0,0)]
. . #     →      . 2 .     →      pop(0,0) push(0,1),(1,0) ...
# . .            . . .            거리는 레벨 = 칸 위 숫자
```

원칙: (1) 입력은 손으로 따라갈 만큼 작게, (2) 변하는 변수만 열로, (3) 각 step의 *판정과 그 결과*를 같이 보여 "왜 이 분기로 갔는지" 드러내기. 사용자가 "그림으로 더 보고 싶다"고 하면 그때만 HTML 시각화 파일을 만들어 열어줘도 된다(기본은 ASCII).

<br><br>

## 제출 전 로컬 검증 (phase 7)

사용자가 가져온 **본인 코드**를 임시로 저장해 엣지케이스를 돌려본다. 정답을 만들어 주는 게 아니라 사용자 코드의 약점을 드러내는 용도다.

```bash
# 사용자 코드를 임시 파일로 (Read로 받은 내용을 Write)
#   /tmp/algo/Main.java
cd /tmp/algo && javac Main.java && echo "5\n1 3 5 7 9\n..." | java Main
```

- 백준 입력 형식 그대로 stdin으로 흘려 결과를 비교.
- 만들 엣지케이스 체크리스트: 최소·최대 N, 경계값(첫/끝), 중복·동점, 단일 원소, 0/음수(허용 범위 내), 합·곱이 int 초과(→ `long` 필요), 정렬 안 된 입력 가정 위반.
- 컴파일·실행이 환경상 안 되면, 케이스만 제시하고 "이 입력에서 네 코드가 뭘 출력할지 손으로 따라가 봐"로 돌린다.
- **자동 백준 제출은 하지 않는다.** 제출·채점은 사용자 몫(결정 사항).

<br><br>

## 막혔을 때 힌트 게이팅 (노트의 "막혔을 때 힌트 순서" 활용)

답 베끼기를 막기 위한 규칙. 한 번에 한 단계만 연다.
1. 30~40분(또는 사용자가 "막혔다") → 힌트 **한 단계만**.
2. 그래도 안 되면 다음 힌트 한 단계 더.
3. 마지막 힌트까지 봐도 안 풀리면 → 접힌 수도코드를 펼쳐 비교(여기까지가 확인용). 그래도 정답 Java는 사용자가 작성.
4. 풀고 나면 시간복잡도를 다시 계산하고 "이 문제의 신호가 무엇이었는지" 한 줄로 PROGRESS.md 메모에 남긴다.

<br><br>

## 절대 규칙 요약
- 정답 Java 코드를 **통째로 주지 않는다.** 막힌 줄/한두 줄 스니펫까지만.
- 접힌 수도코드·체크포인트는 사용자가 **먼저 시도한 뒤** 비교용으로만.
- 권오흠 톤("왜 옳은가") 먼저, 코드는 Java.
- 매 phase 전환마다 PROGRESS.md 갱신(복원 가능성 유지).
- 단계 건너뛰기 금지(단, 사용자가 안 막혔으면 phase 2~4 생략 가능).
- 백준 제출·채점은 사용자가 한다. Claude는 검증·분석만.
