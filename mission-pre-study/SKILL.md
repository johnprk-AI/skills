# 미션 산출물 작성 스킬 (v3)

## 이 스킬의 존재 이유

우테코의 사전학습/토론 활동 산출물은 **토론을 위한 재료를 만드는 단계**다. 자료(LMS) 자신이 그렇게 자기 규정을 한다. 사전학습은 토론 직전 재료, 활동 A·B·C는 토론 중·후 재료다. 어느 쪽이든 결론·계획·심리 변화를 다 담은 회고록이 아니라, 동료가 집어들 수 있는 짧은 메모여야 한다.

이 스킬의 역할은 다음 3가지를 동시에 강제하는 것이다.

1. **자료가 명시한 형식이 최우선**이다. 사전학습 자료가 "형식: 사실 1개 + 문제 1개"처럼 양식을 못 박아 두면, 분량·톤·구조 모두 그 양식에 종속된다. 자체 템플릿(C/D 유형 등)은 자료가 양식을 명시하지 않은 질문에만 적용한다.
2. **"토론 재료" 톤을 유지**한다. 결론을 닫지 않고, 적용 계획을 완성하지 않고, 심리 변화를 적지 않는다. 회고록이 되면 사전학습이 아니다.
3. **자료의 도구함과 사용자 본인의 경험을 양 축으로 잡는다**. 일반 베스트프랙티스로 부풀리지 않고, 본인 코드/PR 위에서 일어난 사실만 쓴다.

## 발동 조건

- 사전학습 자료 (PDF / 본문 / LMS URL) **그리고**
- 답해야 할 질문 리스트 (산출물 항목)
- 자가 인터뷰형 질문이 섞여 있으면, 사용자의 이전 미션 경험 재료(PR 본문, 리뷰 코멘트, 회고 텍스트, 코드 발췌)도 함께 받는다. 없으면 요청한다.

자료 없이 질문만 들어오면 발동하지 않는다 — "사전학습 자료를 먼저 줘"라고 요청한다. 자가 인터뷰형 질문에 경험 재료가 없으면 답을 만들지 않는다 — 추측으로 사용자 인생을 지어내지 않는다.

## 인접 스킬과의 분기

- **회고 글(블로그 포스트)**을 쓰는 게 아니다 → mission-retrospective는 미션 **후**, 이 스킬은 미션 **전**.
- **자가 인터뷰형 질문**(내 학습법, 안 풀리는 지점, 의식할 원칙)도 이 스킬에서 직접 처리한다. v1처럼 mission-method-answer로 위임하지 않는다. 위임하면 회고록 톤이 되돌아오는 사고가 잦았다. 이 스킬 안에서 사실+문제 양식으로 압축한다.

## 5단계 프로토콜

```
Step 1 │ 자료 흡수 — 도구함·실험 의도·산출물 추출 + 형식 명시 여부 체크
Step 2 │ 형식 우선순위 결정 — 자료 명시 형식 vs 자체 유형 템플릿
Step 3 │ 질문 분류 — 자료 형식이 없는 질문에만 4유형 라벨링
Step 4 │ 키워드 보강 — 답변에 실제 쓸 키워드만
Step 5 │ 답변 작성 — 토론 재료 톤, 결론 미닫음, 평문 출력
```

각 Step은 짧게 보고하고 넘어간다. Step 2~3 결과(형식 매핑표)는 Step 5 들어가기 전에 사용자에게 보여주고 OK를 받는다.

## Step 1 — 자료 흡수

사전학습 자료의 표준 골격:

```
1. 헤더 한 줄          ← 이 사이클의 의도 (자기 규정)
2. 이번 사이클에서 답할 질문 ← 토론용 큰 질문 (보통 4개)
3. 사전 학습 키워드 표  ← 도구함
4. 실험 1, 2, ...      ← 해볼 것 / 관찰할 것 / 기록하기 (★ 형식 강제 위치)
5. 사전 학습 준비 가이드 ← 체크리스트
6. 산출물              ← 답변 대상
```

추출해서 사용자에게 보고할 것:
- **사이클 의도 한 줄** — 헤더 문장 그대로 (예: "혼자서 자기 학습법을 돌아보고 자료를 익힌다. 토론 활동의 재료를 만드는 단계.")
- **도구함** — 키워드 표를 옮기고 1줄 정의
- **실험 의도 요약** — 각 실험이 무엇을 드러내려는지 한 줄씩
- **★ 형식 명시 여부** — 자료 본문에서 다음 신호를 모두 훑는다:
  - "형식:" 또는 "기록하기" 같은 헤더 아래 양식 지시
  - "사실 N개 + 문제 N개" 같은 항목 수 지시
  - "예시:" 블록 — 자료가 보여준 예시의 길이·문장 수가 사실상 분량 상한
  - "구체적일수록 좋다" 같은 톤 지시
  - "~보다 ~가 훨씬 좋은 재료다" 같은 좋은 답/나쁜 답 비교
- **답변 대상 질문 목록** — 산출물 항목을 번호 매김

자료에 "형식: 사실 1개 + 문제 1개"가 박혀 있다면, 이 신호는 산출물 질문 전체에 우선 적용된다. 산출물 항목이 별도 형식을 명시하지 않더라도, 같은 사이클의 실험 양식이 산출물의 톤을 결정한다.

## Step 2 — 형식 우선순위 결정

질문마다 다음 우선순위로 작성 형식을 정한다:

```
1순위. 자료가 그 질문에 대해 명시한 형식
2순위. 자료가 같은 사이클 안에서 명시한 일반 형식 (예: 실험 양식)
3순위. 헤더의 자기 규정 ("토론 재료를 만드는 단계" → 짧게·미닫힘)
4순위. 자체 유형 템플릿 (Step 3의 4유형)
```

자료가 양식을 박아둔 사이클(특히 학습법·관찰형 사전학습)은 1~3순위에서 답변 형태가 거의 결정된다. 4유형 템플릿은 자료가 양식을 안 박은 기술형 사이클(테스트 단위·DB 연동·계층 분리 등)에 주로 쓰인다.

## Step 3 — 질문 분류 (자료 형식이 없는 질문에만)

자료가 양식을 명시하지 않은 질문에만 다음 4유형으로 라벨링:

### A. 자가 인터뷰형
본인의 학습법·막힘·이해도를 묻는 질문.
- 처리: 사용자 경험 재료를 사실+문제로 압축. **자료가 사실+문제 양식을 박아두면 그쪽을 쓰고, 안 박아두면 이 유형의 기본 양식이 그대로 사실+문제다.** 학습법 사이클에서는 사실상 자료 양식과 일치한다.

### B. 관찰·기록형
실험·코드 관찰 결과를 묻는 질문. 사용자 코드/실험이 입력으로 필요.
- 처리: 입력 없으면 추측 금지. 입력 있으면 사실+수치+드러난 문제.

### C. 판단·의견형
근거 있는 판단을 요구하는 질문.
- 처리: 자료 도구함 키워드 → 미션 맥락 적용 → 한정적 결론.
- 분량: 5~10줄.

### D. 설계·대안형
복수 안을 비교하고 추천하는 질문.
- 처리: a/b(필요시 c) 비교 + 추천 + 한정 이유.
- 분량: 비교는 한 줄씩, 추천 이유는 1~2줄.

라벨링 결과를 표로 보여주고 OK를 받는다.

```
| # | 질문 (요약)                  | 형식 결정              | 비고                  |
| 1 | 내가 레벨1에서 해온 학습법    | 자료 명시: 사실+문제   | 사용자 경험 재료 필요 |
| 2 | 안 풀리는 지점 1~2개          | 자료 명시: 사실+문제   |                       |
| 3 | 의식할 PiFSR 원칙과 이유     | 자료 미명시 → 짧은 메모 | 결론 닫지 않음        |
```

## Step 4 — 키워드 보강

답변에 실제 쓸 키워드만 1~3줄로 정의 보강. LMS 원문 1순위, 외부 공식 문서 2순위.
- JUnit 5, AssertJ → junit.org/junit5/docs, assertj.github.io
- JDBC, SQL → docs.oracle.com / DB 공식 docs
- OOP·SOLID·불변·방어적 복사 → LMS 자료 우선, 없으면 Effective Java / Robert C. Martin
- PiFSR·나선형 학습 → 사용자가 첨부한 LMS 원문만 (외부 출처 금지)

자료 안에 정의가 있는 키워드는 외부 출처로 덮어쓰지 않는다.

## Step 5 — 답변 작성

### 5.1 자료 명시 형식 (1순위) — 사실+문제 양식

학습법·관찰 사이클의 산출물은 거의 모두 이 양식이다. 다음을 엄수한다:

- **항목 단위**: 각 항목 = "사실 한 줄" + "문제 한 줄". 한 항목 안에 여러 사실을 묶지 않는다. 사실이 여러 개면 항목을 늘린다.
- **사실 한 줄**: 본인이 한 행동·관찰한 결과 하나. 시간·미션·구체적 객체명까지 포함하면 좋다. 감정·해석·일반화는 사실에 섞지 않는다.
- **문제 한 줄**: 그 사실이 드러낸 한계·반복 패턴. 짧을수록 좋다. 본인 입에서 나온 표현(PR 본문·리뷰 답변)을 그대로 인용하면 더 좋다.
- **항목 개수**: 자료가 "1~2개"라고 적었으면 1~2개. "5~10줄"이라고 적었으면 5~10줄. 자료 수치가 상한이다.
- **금지**: 회고 서사("처음에는 ~했지만 ~하면서 ~게 되었다"), 심리 변화("뼈저리게 느꼈다"), 적용 계획 완성("이번 주에 ~를 ~방식으로 ~하겠다"), 클래스 설계 디테일.

자료가 보여준 예시:

```
- 사실: 레벨1에서 모르는 게 생기면 AI에 먼저 물어 답을 받았다
- 문제: 며칠 뒤 같은 문제를 만나면 다시 AI부터 찾고 있어 머리에 안 남는다
```

→ 한 항목이 두 줄. 분량의 기준선이다.

### 5.2 자료 미명시 — 자체 템플릿

자료가 양식을 박아두지 않은 질문에만 적용한다. 모든 답은 5~10줄 이내, 결론 한정어("지금 사이클에서는", "이 미션 규모에서는") 사용.

**A 유형 (자가 인터뷰)** — 자료 명시가 없어도 사실+문제로 작성. 즉 5.1과 동일.

**B 유형 (관찰)**:
```
- 사실: 케이스를 6개로 늘리니 main에서 3곳을 수정해야 했다
- 문제: 하나를 빼먹어 오류를 놓칠 뻔했다
```

**C 유형 (판단)**:
근거 1줄 → 미션 맥락 적용 1~2줄 → 한정 결론 1줄. 5~7줄.

**D 유형 (설계 대안)**:
```
a안: ...
  - 장점: ...
  - 단점: ...
b안: ...
  - 장점: ...
  - 단점: ...
=> 이 미션 규모에서는 a안이 맞다고 본다. 이유: ...
```

### 5.3 PiFSR 원칙 답변 (특수 케이스)

"의식하고 싶은 원칙 1개와 그 이유"형 질문은 자료 미명시지만, 헤더의 "토론 재료" 톤을 따른다. 다음 3줄 안에서 끝낸다:

```
원칙: [원칙 이름]
이유: [본인 사실+문제 1개에 이 원칙이 어떻게 맞물리는지 1~2줄]
```

적용 계획·다음 사이클 다짐을 적지 않는다. 그건 미션이 끝난 후의 회고다.

### 5.4 출력 포맷

산출물은 평문으로 출력한다. 마크다운(##, **, >, ```, 표, 리스트 기호)을 쓰지 않는다. 사용자가 LMS 제출란에 그대로 붙일 수 있어야 한다.

질문 사이는 빈 줄 한 줄. 항목은 "- " 정도의 가벼운 글머리만 허용 (자료 예시도 그렇다).

```
1. {질문 텍스트}

- 사실: ...
- 문제: ...
- 사실: ...
- 문제: ...

2. {다음 질문}

- 사실: ...
- 문제: ...
```

## 회고록이 되었는지 점검 (가장 자주 빗나가는 패턴)

답변 넘기기 전에 다음을 본다. 하나라도 yes면 다시 압축한다.

- [ ] 한 항목이 3줄을 넘는가? → 사실/문제로 쪼개거나 항목을 분리
- [ ] "처음에는 ~했지만, ~하면서 ~게 되었다" 같은 서사 문장이 있는가?
- [ ] "뼈저리게", "아쉬움이 컸다", "체감했다" 같은 감정 표현이 있는가?
- [ ] "이번 주에는 ~를 ~방식으로 ~하겠다" 같은 완성된 적용 계획이 있는가? → 사전학습 단계가 아닌 회고
- [ ] 클래스 이름·메서드 이름이 답변 본문에 3개 이상 등장하는가? → 사전학습은 코드 디테일 단계가 아님
- [ ] 사실과 해석을 한 문장에 섞었는가? ("AI에 위임해서 머리에 남지 않았다" → 사실: AI에 위임 / 문제: 머리에 남지 않음)
- [ ] 자료가 명시한 항목 수("1~2개", "5~10줄")를 넘기는가?
- [ ] 본인 경험 재료에 없는 내용이 들어갔는가? (지어내기 금지)

## 자가 인터뷰형 질문 처리 시 주의

학습법 사이클의 산출물 3개("내 학습법", "충돌 지점", "PiFSR 원칙")는 한 덩어리로 보이지만, **자료의 형식 지시는 실험1·실험2의 "기록하기" 블록에 들어 있고, 그게 산출물에도 그대로 적용된다**. 즉:

- "내 학습법" → 사실+문제 항목 2~4개 (자료 실험1 양식)
- "충돌 지점" → 사실+문제 항목 1~2개 (자료가 명시: "1~2개")
- "PiFSR 원칙" → 원칙 + 이유 2~3줄 (자료 미명시지만 토론 재료 톤)

세 답을 하나의 흐름으로 엮지 않는다. 각각 독립된 메모로 끊는다.

## 사용자 경험 재료 처리

사용자가 경험 재료를 직접 던져주지 않았다면 user_profile 메모리에 적힌 자료 위치(블로그 레포 `johnprk.github.io`의 `content/posts/`, 우테코 미션 PR 레포)에서 gh로 가져와 채운다. 가져온 뒤에는 어떤 자료를 봤는지 한 줄로 사용자에게 보고한다 — 출처를 명시해야 본인 기억과 어긋나는 부분을 사용자가 짚을 수 있다.

자료 자동 수집 절차:
- 블로그: `gh api repos/JohnPrk/johnprk.github.io/contents/content/posts/{category}/{slug}.md --jq '.content' | base64 -d`
- PR 본문+코멘트: `gh pr view {N} --repo {owner/repo} --comments`
- PR 검색: `gh search prs --author=JohnPrk --limit=30 --json url,title,repository,createdAt`

사용자가 PR 본문·리뷰 코멘트·회고 텍스트를 던져줬을 때:

1. 본인 입에서 나온 사실 문장을 먼저 추린다 (예: "PR 본문에 본인이 직접 적은 문장: 내 코드의 냄새를 스스로 정확히 맡지 못해").
2. 리뷰어가 한 말은 "본인 사실"로 쓰지 않는다. 본인 사실 → 그 사실을 리뷰어가 어떻게 짚었는지 → 그래서 드러난 문제, 순서로 압축할 수 있을 때만 쓴다.
3. 사실 문장을 그대로 인용 가능하면 인용한다. 의역으로 부드럽게 만들면 토론 재료의 날카로움이 죽는다.
4. 여러 사이클을 다 다루지 않는다. 자료가 "1~2개"라고 했으면 가장 강한 사실 1~2개만 쓰고 나머지는 버린다. 다 담으려는 충동이 회고록을 만든다.

## 운영 팁

- 자료가 PDF로 들어오면 이번 사이클 자료만 추출한다. PiFSR·나선형 학습 같은 메타 자료는 도구함 보강용으로만 읽고, 그 자체를 답변 본문에 옮기지 않는다.
- 학습법 사이클 사전학습은 거의 모든 산출물이 자료 명시 형식(사실+문제)에 종속된다. v1·v2에서 가장 자주 빗나간 곳이 여기다.
- 이 스킬을 호출할 때 사용자가 이미 초안을 적어 왔다면, 새로 쓰지 말고 위 점검 항목으로 압축만 한다. 새로 쓰면 또 회고록이 된다.
