# 미션 회고 작성 스킬

## 이 스킬의 존재 이유

자료(코드·PR·리뷰)만 받고 바로 글을 쓰면 반드시 세 가지가 망가진다.

1. **경험 날조** — 사용자가 실제로 해보지 않은 시도 과정을 일반화로 끼워 넣게 됨
2. **과잉 해석** — 사용자가 아직 체화하지 못한 개념을 "깨달았다"로 서술하게 됨
3. **뒤늦은 구조 변경** — 초안 다 쓴 뒤에 RC 라운드 분리·감정 아크가 들어와 전면 재작성됨

자료와 글 사이에는 "사용자의 실제 경험과 감정을 끌어내는 인터뷰 단계"가 있어야 한다. 이 스킬의 본질은 그 사이 단계를 강제하는 것.

## 5단계 프로토콜

```
Phase 0 │ 자료 수집   — 사용자가 한 번에 투입
Phase 1 │ 탐색 질문   — 3~4개씩 반복 인터뷰
Phase 2 │ 구조 합의   — 아웃라인/메타 테마/감정 아크
Phase 3 │ 섹션 초안   — H1 한 덩이씩 순차 작성
Phase 4 │ 퇴고        — 라인 단위 피드백
```

각 Phase는 앞 단계가 끝나기 전에 시작하지 않는다. 특히 **Phase 2 합의 없이 Phase 3로 넘어가지 않는다.** Phase 2 → 3 경계에는 강제 멈춤(`/clear`)이 있다 — Phase 2 섹션 참고.

## 파일 구조 컨벤션

회고 작업이 시작되면 동일 디렉토리에 네 파일을 유지한다.

```
회고-작업상태.md  ← 복귀 시 가장 먼저 읽는 체크포인트. 현재 Phase / 다음 할 일 / 합의 사항 / 실제 사용자 표현
회고-재료.md      ← Phase 1 인터뷰 원본 답변을 Q/A 포맷으로 누적
회고-구조.md      ← Phase 2에서 합의한 아웃라인/테마/감정 아크
회고-초안.md      ← Phase 3~4에서 쓰고 있는 글 본체
```

`회고-재료.md`는 **절대 각색하지 않는다**. Phase 4에서 왜곡 점검용 대조 자료로 쓴다.

### 작업 위치 결정 (Phase 0 전에)

#### 🚨 게이트 0 — 폴더·파일을 만들기 전에 대상부터 확인받는다

스킬이 발동되면 mkdir / Write를 하기 **전에** 반드시:

1. `~/Desktop/리소스/회고/CLAUDE.md`의 **블로그 글 매핑 표 + 시리즈 순서**를 읽는다.
2. 사용자에게 한 줄로 확인받는다: "이번 회고 대상 = **<미션명> (<PR 링크>)**, 작업 폴더 = `<슬러그>/` 맞아?" 시리즈 순서상 앞에 빠진 회고가 있으면 그것도 같이 짚는다.
3. 사용자 OK 전에는 폴더도 파일도 만들지 않는다.

**Why:** 2026-06-09에 세션이 열린 폴더(level2-wrapup)를 회고 대상으로 추정하고 확인 없이 스캐폴딩을 만들었다가, 실제 대상(member-cycle-1)이 따로 있어서 55분을 쓰고 rm -rf로 되돌린 사고. **세션이 열린 폴더는 대상의 근거가 아니다.** 사용자는 아무 폴더에서나 세션을 연다.

회고 작업 파일 4종은 **항상 `~/Desktop/리소스/회고/<슬러그>/` 아래에 만든다.** 이 디렉토리가 사용자의 회고 자료 표준 보관소다. 현재 작업 디렉토리(미션 코드 레포 등)에는 회고 파일을 만들지 않는다 — 코드 레포를 더럽히지 않기 위함이다.

규칙 (`~/Desktop/리소스/회고/CLAUDE.md`에 기록된 워크스페이스 규칙과 동일):

1. **단일 평탄 구조.** 카테고리 분기(코딩 미션 / AI 협업 등) 없이 슬러그 폴더만 나란히 둔다.
2. **폴더명 = 대응되는 블로그 글의 영문 슬러그.** 예: `janggi-cycle-2`, `roomescape-admin-cycle-1`, `tokens-100m`. 슬러그는 사용자와 합의해서 정하되, 합의 전이라도 임시 슬러그로 폴더를 만들어 시작하고 확정되면 폴더명을 맞춘다.
3. **해당 슬러그 폴더가 없으면 만들고**, 그 안에서 4파일(`회고-작업상태.md`, `회고-재료.md`, `회고-구조.md`, `회고-초안.md`)을 생성·갱신한다.
4. 필요하면 폴더의 `CLAUDE.md`에 글 개요·PR 링크·진행 Phase를 기록한다 (권장, 필수 아님).

`~/Desktop/리소스/회고/CLAUDE.md`에는 슬러그 폴더 ↔ 블로그 경로 매핑 표가 있다. **폴더 생성과 매핑 표 행 추가는 한 동작이다** — 폴더를 만들었는데 표를 안 고쳤으면 그 작업은 안 끝난 것. 글 상태가 바뀔 때(초안 완성·게시)도 비고 칸을 같이 갱신한다. (2026-06-09에 이 줄이 무시돼 표가 stale로 남았고, 그 stale 표가 다음 세션의 게이트 0 오판으로 이어졌다.)

### 복귀 프로토콜 (토큰 리셋 / 세션 재개 시)

이 스킬이 새 세션에서 발동되면 **반드시 이 순서로 파일을 읽는다**:

1. `회고-작업상태.md` — 현재 Phase, 다음 할 일, 합의된 메타 테마/구조/톤, 금지 사항 복원
2. `회고-구조.md` — 아웃라인 세부
3. `회고-재료.md` — 인터뷰 원본 (필요 시)
4. `회고-초안.md` — 지금까지 쓴 본체 + 꼭지별 완료 상태

`회고-작업상태.md`에 기록된 합의 사항은 **사용자 재합의 없이 변경하지 않는다**. 새 Phase 2를 다시 열지 않는다.

Phase 2~3 사이에 처음 `회고-작업상태.md`를 생성할 때는 다음을 최소한 담는다:
- 현재 Phase
- 다음 구체적인 할 일
- 합의된 메타 테마 한 문장
- H1 꼭지 목록과 순서
- 꼭지별 감정 레이블
- 의도적으로 **안 쓰기로** 한 것들
- 사용자의 실제 표현(그대로 살릴 인용구) 목록
- 꼭지별 완료 체크리스트

## Phase 0 — 자료 수집

사용자에게 다음을 한 번에 요청한다.

- 대상 미션 개요 한 줄
- PR URL (GitHub MCP로 자동 수집 — 아래 참고)
- 관련 Slack/DM/이슈 대화
- 개인 메모·일기가 있다면 그것도
- 이전 미션 회고가 있다면 연결점 찾기용으로 함께

### PR 자료 수집 — GitHub MCP 우선

PR URL을 받으면 **WebFetch 대신 GitHub MCP 도구로 가져온다.** 순서:

1. PR 본문 + 메타 정보 (`get_pull_request`)
2. 리뷰어별 리뷰 코멘트 전체 (`get_pull_request_reviews`)
3. inline 코멘트 (코드 라인 위치 포함) (`list_pull_request_review_comments`)
4. 일반 이슈 코멘트 (`list_pull_request_comments`)

RC(재검토 요청) 라운드가 여러 번이면 각 라운드별로 리뷰를 구분해서 `회고-재료.md`에 기록한다.  
GitHub MCP를 쓸 수 없는 환경이면 WebFetch로 fallback.

자료 투입이 끝나면 바로 Phase 1로 넘어간다. **자료 기반으로 요약이나 초안을 시도하지 않는다.**

## Phase 1 — 탐색 질문 (반복 인터뷰)

### 진입 전 — 3줄 컨텍스트 먼저 받기

4축 질문에 들어가기 **전에** 사용자에게 다음 3줄을 먼저 요청한다. 이 3줄이 없으면 4축 질문이 일반론으로 흐르고, 글의 독자상이 후반에 흔들린다.

1. **이 미션 1줄 요약** — 무엇을 만드는 미션이었는지
2. **사용자 현재 상태** — 이 글을 쓰는 시점에 사용자가 어디까지 와 있는지 (예: 머지 직후 / 다음 미션 진행 중 / 한참 지난 회상)
3. **이 글의 독자상** — 누가 읽기를 기대하는지 (동료 크루 / 리뷰어 / 미래의 본인 / 외부 블로그 독자)

3줄을 받아 `회고-작업상태.md` 상단에 박아두고, 이후 모든 Phase에서 이 3줄을 기준으로 판단한다.

⚠️ **시리즈·인접 미션 사실은 미확인으로 취급한다.** 1번(미션 요약)에 "전 미션은 X였다" 같은 *다른 미션·사이클에 대한 사실 주장*이 섞이면, 미션 *이름*에서 내용을 추정하지 말고(예: `member` 미션 → "로그인"으로 단정 X) 재료·PR·사용자 확인으로 검증한다. 확인 전에는 작업상태에 `(미확인)`을 달아 두고 초안에 사실로 굳히지 않는다. (2026-06-15: "전 미션=로그인"이 미확인 채로 작업상태에 박혀 서론까지 전파된 사고.)

### 규칙

- 한 번에 **3~4개**만 질문한다. 10개씩 몰아 묻지 않는다.
- 답을 받으면 그 답을 근거로 다음 3~4개를 조립한다.
- 답변은 즉시 `회고-재료.md`에 `## Q: ...` / `A: ...` 포맷으로 누적한다.
- 답이 모호하거나 짧으면 같은 축으로 재질문한다 ("더 구체적으로 어떤 순간이었어요?").

### 질문 4축

턴마다 이 네 축을 고루 순환시킨다. 한 축만 계속 파지 않는다.

**① 사실 확인 축** — 경험 날조 방지
- "이 추상화는 설계해놓고 구현한 거예요, 구현하다 드러난 거예요?"
- "실제로 어떤 방식을 먼저 시도했고, 어디서 막혔어요?"
- "이 코드 구조는 리뷰 전부터 있던 거예요, 리뷰 후에 바뀐 거예요?"
- "이 패턴을 적용하게 된 결정적 계기가 뭐였어요?"

**② 감정 추적 축** — 감정 아크 재료 확보
- "RC 알림 열었을 때 첫 반응이 뭐였어요?"
- "가장 아팠던 리뷰 한 줄을 꼽으면?"
- "PR 올리기 직전 기분은요? 자신 있었어요, 긴장했어요?"
- "머지된 순간 어떤 기분이었어요?"

**③ 이해도 체크 축** — 과잉 해석 방지
- "이 개념, 지금 어디까지 이해됐다고 느껴요?"
- "아직 잘 모르겠는 부분 솔직히 꼽으면?"
- "이 원칙을 다음 미션에서 자신 있게 적용할 수 있을 것 같아요, 아직 감이 덜 왔어요?"
- "리뷰어가 한 말 중에 '말은 알겠는데 왜 그런지는 아직 모르겠는' 부분 있어요?"

**④ 독자 의도 축** — 주제 방황 방지
- "이 글을 다 읽은 사람이 뭘 가져갔으면 좋겠어요?"
- "타겟이 동료 크루예요, 리뷰어예요, 미래의 본인이에요?"
- "이 회고가 다른 미션 회고랑 구별되게 하는 한 가지가 뭘까요?"

### Phase 1 종료 조건

다음이 모두 확보되면 Phase 2로 넘어간다.

- [ ] 주요 에피소드마다 **실제 시도 순서**가 기록됨 (날조 없이 복원 가능)
- [ ] 각 에피소드·RC 라운드에 **감정 레이블** 최소 하나 붙음
- [ ] 핵심 개념마다 **사용자 이해도 레벨** 체크됨 (완전히/부분적으로/아직)
- [ ] **독자상·전달 메시지**에 대한 사용자 답이 있음

## Phase 2 — 구조 합의

`회고-구조.md`에 다음 다섯 항목을 채워서 제시하고, 사용자의 OK를 받는다.

1. **전체 꼭지 분할** — H1 기준 몇 덩이로 나눌지 (예: `PR 제출 전 / 1차 RC / 2차 RC / 3차 리뷰 / 회고`)
2. **관통하는 메타 테마 한 문장** — 글 전체를 꿰는 단 하나의 문장
3. **각 꼭지의 감정 상태 레이블** — (예: `자신감 → 당혹 → 조심 → 각성 → 성찰`)
4. **에피소드 우선순위** — 깊게 팔 3~4개 vs 짧게 언급할 것들
5. **서론 진입점과 결론 착지점** — 어디서 시작해서 어디로 내릴지

사용자가 OK할 때까지 수정을 반복한다. **OK 전에 Phase 3로 넘어가지 않는다.**

### 🚨 Phase 2 → 3 경계 — 강제 멈춤 (/clear)

구조 OK가 나오면 **같은 세션에서 Phase 3을 시작하지 않는다.** 합의 내용을 `회고-작업상태.md`에 저장한 뒤 한 줄로 안내하고 멈춘다: "구조 합의 끝. `/clear` 하고 오면 복귀 프로토콜로 Phase 3 시작할게."

- 복귀 프로토콜이 4파일에서 컨텍스트를 복원하므로 잃는 것이 없다.
- 예외: 사용자가 "그냥 이어서 해"라고 명시하면 진행한다. 그 경우에도 꼭지 직전 3점검(Phase 3)은 그대로 수행한다.

**Why:** 2026-06-09 세션이 Phase 1→3을 11시간 한 세션으로 끌었고, 중간 /compact가 스킬 본문을 요약으로 압축하면서 문체 금지 룰이 쓸려나갔다. 그 8시간 뒤 Phase 3 작성에서 "출처가 단단하지 않았다" 류 누수가 났다. 토큰 비용 문제이기 전에 품질 문제다.

### 헤딩 레벨과 번호 규약

- **본문 메인 섹션은 `###`(H3)을 기본으로 사용한다.** `##`(H2)은 글 전체를 큰 분기(예: PR 제출 전 / 1차 RC / 2차 RC / 회고)로 또렷이 가를 때만. 한 글에 H2가 5개 이상 쌓이면 거의 잘못 고른 것.
- **소제목에 `1.`, `2.`, `3.` 같은 일련번호를 붙이지 않는다.** 번호는 순서를 강제하지만 회고는 시간순·주제순이 섞이는 게 보통이라 번호가 어색하다. 사용자 옛 회고들도 번호 없이 명사구·완결문 헤딩만 씀(예: `근데 이게 페어 프로그래밍인가?`, `리뷰어가 답했다.`, `머지 이후 남겨진 것들`).
- **헤딩은 명사구 또는 인용·짧은 완결문.** "충격", "답답함" 같은 단어 단독 정서어를 헤딩에 박지 않는다(아래 문학체 금지 룰과 같은 결).

### 사이클 회고 표준 구조

우테코 사이클 미션 회고는 7편 분석 결과 다음 4단 구조가 표준이다(블랙잭/장기/방탈출 사이클 전부 이 모양):

```
서론 (1~2 단락, 짧음)
  - 미션 진입 직전 본인 상태·기대·맥락
  - 사실 위주. 감정 수식은 한 문장 이내
  ↓
본문 꼭지들 — ★시간순(연대기) 배열이 기본. 주제별로 묶지 말 것.
  ① 미션 전 내 생각 (사전학습·토론·페어프로그래밍, 진입 stance)
  ② 미션 하면서 든 생각 (PR 제출 시점 설계 + 리뷰어에게 처음 던진 질문/근거)
  ③ 중간에 바뀐 부분 (리뷰어 피드백 + 외부 입력: 강연·공식문서 등)
  ④ 수정한 부분 (실제 코드·설계 변경)
  ⑤ 또 바뀐 부분 (추가 리뷰 라운드, 남은 합의 / 미해결)
  ⑥ 마무리 및 정리
  - 각 꼭지: 리뷰어 코멘트 `>` 인용(2~5줄) + 본인 반응·분석 + (가능하면) 코드 before/after
  - 같은 주제(DDD/REST 등)가 여러 시점에 걸치면 한 곳에 모으지 말고 ②③④⑤에 시점대로 흩뿌린다
  ↓
결론 ("마치며" / "비로소 보이는 것" / "사이클N을 마치며")
  - 거대 일반화 X
  - 남은 숙제·부채 명시
  - 짧은 한 문장으로 닫음
  ↓
### 참고 (사이클 미션 한정)
  - 🚀 사이클N - 미션 (...) PR : <URL>
  - 이전 사이클 회고 링크 (있으면)
```

**🚨 본문 꼭지 순서 = 시간순 (사용자 필수 규칙, 2026-06-10)**
- 사용자가 명시: "아예 시간의 순서대로 해야할 것 같아. 그 당시의 사건들이 깨달았던 일들이 많아서." + "앞으로도 회고는 무조건 이렇게." → **모든 회고는 위 ①~⑥ 연대기 골격으로 시작한다.**
- 주제별 분할(꼭지1=DDD 전체 arc, 꼭지2=REST 전체 arc) 금지. 한 미션에서 깨달음이 시간차로 쌓였으면, 그 인과(무엇을 듣고 → 무엇이 바뀌고 → 무엇을 고쳤는지)가 시간선에서 드러나야 한다.
- ①~⑥은 고정 헤딩이 아니라 골격. 실제 헤딩은 명사구·인용으로 짓되 배열만 시간순을 지킨다. 사건이 적으면 ②~⑤를 합쳐도 되지만 순서는 유지.

**예외**:
- 대화형 회고(석식챗·커피챗 등)는 본문 꼭지 자체가 대화 흐름이고, 외부 PR이 없으므로 `### 참고` 섹션 생략. 본문 안에 인라인 링크.
- 비-사이클 회고(데모데이·1주차 등)는 PR 링크 대신 산출물 링크.

**금지**:
- 결론 꼭지가 끝나기 전에 새로운 사건/리뷰어 인용을 끌어오지 않는다. 결론은 닫는 자리.
- "다음 사이클에서는 ~할 것이다" 류 미래 예고는 한 줄 이내. 다음 사이클 회고를 미리 쓰지 않는다.

## Phase 3 — 섹션 초안 (한 덩이씩)

### 규칙

- 한 번에 **H1 한 꼭지만** 쓴다.
- 한 꼭지 쓰면 멈추고 피드백을 받는다.
- **🚨 쓴 꼭지는 전문을 채팅에 그대로 펼친다.** 파일(`회고-초안.md`)에만 반영하고 채팅에서는 요약·구조 설명·검토 의견만 내놓지 않는다. 사용자는 채팅에 펼쳐진 *실제 문장*으로 검수한다. 요약·평가만 보여주면 사용자가 문장 단위로 확인·교정할 수 없다. 파일이 (다른 세션·사용자 손으로) 이미 채워져 있던 경우에도, 추정·요약으로 넘기지 말고 그 전문을 채팅에 펼쳐 검수받은 뒤 다음으로 넘어간다. (2026-06-17: ④ 꼭지를 파일에만 두고 채팅엔 평가만 내놨다가 "내용을 다 보여줘야 내가 확인을 하지" 지적.)
- 피드백 반영 후 OK 받으면 다음 꼭지로 넘어간다.
- **전체를 한 번에 써서 보여주지 않는다.**

### 작성 규칙

- 해당 꼭지의 감정 레이블(Phase 2)에 맞는 온도로 문장을 쓴다.
- 문단은 2~3문장 단위로 잘게 자른다.
- **한 꼭지를 시작하면 끝까지 한 호흡으로 쓴다.** 중간에 "여기까지 OK?", "이 톤 맞아요?" 같은 확인 질문을 끼워 넣지 않는다. 사용자의 짧은 응답이 "계속해" 신호로 처리되면서 잘못된 방향이 그대로 굳어진다. 피드백은 꼭지가 다 끝난 뒤에 받는다.

### 🚨 꼭지를 보여주기 직전 — 필수 3점검 (매 꼭지, 생략 금지)

긴 세션·/compact 이후에는 이 파일의 톤 룰이 컨텍스트에서 흐려진다. 컨텍스트에 남아 있다고 믿지 말 것 — compact가 지나갔으면 이미 요약돼 사라졌다. 각 꼭지를 사용자에게 보여주기 직전에 매번:

1. **톤 섹션 재독** — 이 파일의 "문학체·시적 수식" + "사용자 톤 마커" 섹션을 Read 도구로 다시 읽는다 (`~/.claude/skills/mission-retrospective/SKILL.md`).
2. **재료 축 커버 대조** — `회고-재료.md`에서 이 꼭지의 시기에 해당하는 Q/A를 다시 읽고, 사용자가 비중 있게 말한 주제가 초안에서 빠졌는지 확인한다. (2026-06-09: 인터뷰에서 길게 말한 REST가 초안에서 통째로 빠져 "너무 빈약한데?" 지적을 받음.)
3. **lint 경고 즉시 반영** — `회고-초안*.md` 저장 시 자동 lint 훅(retro-lint.py)이 경고를 돌려주면, 사용자 인용 원문이 아닌 한 보여주기 전에 고친다.

### 문학체·시적 수식 — 강제 금지와 의심 신호 분리

사용자 옛 회고 7편을 분석해보면, 일부 정서 수식(당혹·찜찜함·후련함 같은 짧은 명사 + "었다")은 자연스럽게 깔려 있다. *모든 정서 수식을 박멸하면 사용자 톤이 죽는다.* 그래서 두 단계로 나눈다.

#### 🚨 강제 금지 (작성 시점에 절대 X)

1. **사물·감정의 의인화** — "어색함이 먼저 왔다", "충격이 나를 멈추게 했다", "막막함이 더 크게 다가왔다", "그 자리에서 슬럼프가 왔다"
   → 감정명사가 *주어로 움직임*을 갖는 형태. 사실로 다시 쓴다.
2. **마음·감정의 풍경 묘사** — "마음 한구석에는 ~이 그대로 남아 있었다", "마음에 남았다", "어디로 갔는가"
   → 마음·감정을 *공간*으로 다루는 비유.
3. **헤딩 자체에 정서어 단독 사용** — "...마주친 충격", "...라는 익숙한 답답함", "비 오는 날, 안 친한 조합" 류
   → 헤딩은 명사구·인용·짧은 완결문. 정서 라벨 X.

이 3가지는 사용자가 명시적으로 "문학적으로 써줘"라고 한 경우에만 풀린다.

#### ⚠️ 의심 신호 (본문에 한 번 등장은 OK, 같은 단락에 두 개 이상 겹치면 정리)

4. **시간·대화·자리의 자율적 흐름 묘사** — "그게 그대로 길게 이어진 대화가 됐다", "자리가 그렇게 지나갔다"
5. **추상명사 + "라는 것" 류 회상조** — "어떤 안도감 같은 것", "한 장면의 충격이라기보다"
6. **이중 부정·역설 수식** — "사라진 게 아니라 자리 잡았다", "끝난 게 아니라 시작이었다"

이 3가지는 옛 글(blackjack-cycle-1, janggi-cycle-2)에도 *드물게* 등장. 사용자 톤의 일부일 수 있다. 다만:
- 한 글에 5번 이상 반복되면 톤이 인위적으로 흐름 → 정리
- 같은 단락 안에 두 개 이상 겹치면 → 하나로 줄임
- *내용이 이미 드러나 있는데도* 수식이 붙어 있으면 → 삭제

#### 의심 단어 빠른 점검 (응답 보내기 직전)

본문에서 다음 단어들이 *내가 쓴 자리*(사용자 인용문이 아닌)에 있으면 한 번 더 본다:
- **주어로 움직이는 정서명사** — 충격이 ~했다 / 막막함이 ~다가왔다 / 답답함이 ~따라왔다
- **공간화한 마음 표현** — 마음 한구석 / 마음에 남았다 / 마음의 자리

이 둘이 보이면 무조건 의인화·풍경 묘사이므로 → 사실/발언 기반으로 다시 쓴다.

#### 대체 원칙

- 정서를 *말해서 전달*하지 말고, 사실·발언·시퀀스로 *드러내서 전달*한다.
- "~다는 사실이 또렷하게 다가왔다" 류는 → "~를 확인했다" / "~가 보였다" 정도로 평이하게.
- 정 정서를 살리고 싶으면 사용자의 *실제 발언*을 직접 인용(`>`).

### 사용자 톤 마커 — 적극 활용

옛 회고 7편에 일관되게 등장하는 표현들. *문학적 수식 대신 이 마커들을 살려서 톤을 만든다.* 사용자 본인이 자연스럽게 쓰는 어법이라 위조처럼 안 들린다.

**단정 회피 어법** (사용자 톤의 핵심):
- "~에 가까웠다" / "~쪽이었다" / "~인 것 같았다"
- "~다는 생각이 들었다" / "~라는 느낌이었다"
- 예시(blackjack-cycle-1): "충격이라기보다는 확인에 가까웠다"
- 예시(janggi-cycle-2): "~를 알 수 있는 시간이였다는 것이었다"

**자기 검열·강조 마커**:
- "솔직히 말하면" / "사실"
- "곰곰이 생각해보니" / "다시 생각해보니"
- "굳이" (의문/거부 진입어)
- "막상 ~보니" / "막상 ~하자" (예상 vs 현실 차이)

**무지 인정 → 학습 진입** (가장 강한 사용자 톤. janggi-cycle-2 전체):
- "그냥 썼던 것 같습니다"
- "한 번도 들여다본 적이 없었다"
- "선택을 했다고 생각했는데 사실은 선택한 적이 없었다"
- "그게 뭔지 설명할 수가 없었다"

→ 리뷰어가 "왜 X를 선택했나요?" 물었을 때, *이 마커로 진입*한 뒤 공식 문서 학습으로 넘어가는 동선이 사용자 표준이다.

**규칙**:
- 이 마커들을 *내가 만들어 넣지 말고*, 사용자가 인터뷰(Phase 1)에서 실제로 쓴 표현을 그대로 옮긴다.
- Phase 1에서 사용자가 안 쓴 마커를 본문에 넣지 않는다.
- 같은 마커("솔직히 말하면" 등)를 한 글에 3번 이상 쓰지 않는다.

### 리뷰어 질문 응답 패턴 (사이클 회고의 표준 단위)

리뷰어가 "X를 왜 선택하셨어요?" 류 단답형 질문을 던진 자리는, 사용자 회고에서 가장 강한 톤이 나오는 자리. 4단계 동선을 표준으로 한다 (janggi-cycle-2 전체가 전형):

1. **리뷰어 질문 인용** — `>` 블록 한 줄
2. **무지 인정 (단문 1~2문장)** — "그냥 ~했다", "한 번도 들여다본 적이 없었다" 류
3. **공식 문서 기반 메커니즘 정리** — 1~2 단락. 키워드 추적 톤. 출처 명시
4. **본인 코드에 어떤 영향을 줬는지** — 수정 한 줄이든, 그대로 둔 결정이든

이 동선이 흔들리면(예: 무지 인정 빼고 바로 공식 문서로 들어감) 사용자 톤이 인위적이 된다. 4단계가 거의 항상 다 들어간다.

### 결론 톤

마지막 꼭지("마치며" / "비로소 보이는 것" / "사이클N을 마치며") 작성 규칙:

- **거대 일반화·각성 선언 X.** "이 미션이 나를 바꿨다" / "객체지향이 무엇인지 깨달았다" 류 금지.
- **남은 숙제·부채를 같이 명시.** 4/7개 옛 글에 이 패턴 있음. 무엇을 *아직 모르는지*가 결론의 핵심 자료.
- **짧은 한 문장으로 닫는다.** 예: "객체지향을 더 공부하고 싶다. 지금으로서는 그게 이 사이클이 남긴 가장 정직한 문장인 것 같다."
- **단정 회피 어법으로 마무리.** "~인 것 같다" / "~쪽이다" / "~이라는 생각이 든다"

결론에 정서명사("후련함", "벅참") 단독 한 번은 OK. 다만 "후련함을 안고 ~", "벅찬 마음으로 ~" 류 비유는 X.

### "더 길게 / 풍성하게" 요청을 받을 때 — A/B/C/D 옵션 금지

사용자가 "양이 좀 부족한데", "더 풍성하게", "디테일을 더" 같은 요청을 하면 **확장 옵션을 A/B/C/D로 제안하지 않는다.** 옵션을 만드는 순간 skill은 재료에 없는 시도·감정·이해를 메뉴로 진열하게 되고, 사용자는 자연스러워 보이는 것을 골라 본문에 채워 넣는다. 결과는 사실상 위조다.

대신 다음 두 가지 중 하나로 응답한다.

1. **새 재료를 요청한다** — "이 부분을 더 길게 가려면 [구체적으로 어떤 순간/감정/시도]가 필요한데, 그게 실제로 있었어요?"
2. **자리를 비워둔다** — 재료가 없으면 그 꼭지는 짧은 채로 둔다. 짧은 게 정직한 길이라는 신호다.

옵션을 늘어놓는 친절함은 사용자에게 도움이 되는 것처럼 보이지만, 글의 신뢰도를 가장 빠르게 갉아먹는 패턴이다.

## Phase 4 — 퇴고

### 진입 전 경고 — 톤 일치 ≠ 내용 정확

이 skill이 사용자 말투·문체·감정 결을 잘 흉내 낼수록 **위조의 품질도 같이 올라간다**. 톤이 자연스러울수록 사용자는 본문을 통째로 신뢰하고 검증을 건너뛰기 쉽다. 자연스러움은 정확성의 보증이 **아니다**. 톤이 매끄러워 보이는 꼭지일수록 아래 체크리스트를 더 엄격히 돌린다.

### 퇴고 체크리스트

사용자의 라인 단위 피드백을 받는다. 수정할 때마다 점검한다.

**사실 정확성** (매 꼭지 수정 후 필수)
- [ ] 사용자가 말하지 않은 시도 과정을 내가 만들어 넣지 않았나?
- [ ] 사용자가 "아직 모른다"고 한 개념을 "깨달았다"로 쓰지 않았나?
- [ ] 존재하지 않는 다음 미션/사이클을 예고하고 있지 않나?
- [ ] 구조가 Phase 2 합의를 벗어났나?
- [ ] 감정 아크가 합의한 레이블에서 이탈했나?
- [ ] 기술 세부사항이 사용자가 실제로 찾아본 범위를 넘어 딥다이브로 빠지지 않았나?
- [ ] 본문의 기술 식별자(클래스명·메서드명·변수명·enum 값·라이브러리명·설정 키)가 `회고-재료.md`에 실제로 존재하는가?
- [ ] 다른 미션·사이클·시리즈에 대한 사실 주장("전 미션은 X였다" 류)이 재료·PR·사용자 확인에 근거하는가? 미션 *이름*에서 내용을 추정하지 않았나? (2026-06-15: `member`→"로그인" 단정으로 서론까지 오염된 사고)

의심이 들면 **항상 `회고-재료.md`의 원본과 대조.** 그래도 애매하면 사용자에게 확인.

**스타일 — 자주 부풀려지는 패턴**

1. 드라마 수식어 반복 ("당혹이었다", "솔직히 말하면", "~깨달았다") → 내용이 이미 드러나 있으면 삭제
2. 산문 속 3개 이상 나열 → 목록으로 올리기
3. 소제목마다 리뷰어 인용(`>`) 헤드 → 꼭지당 1~2개만
4. 꼭지마다 `---` + bridge 반성 문단 → 클라이맥스·맺음 직전 1~2곳만
5. chronology caveat 남발 → 독자가 정말 헷갈리는 자리에서만
6. 본류와 무관한 과거 회상·"한때 시도했던 것" 문단 → 본류에 힘 실어줄 때만
7. 종속절 겹치기 → 마침표로 끊어 선언형 단문으로
8. 메타 테마·핵심 정리 → 본문에 섞지 말고 단독 문단 + 굵게
9. Phase 1 인용구 억지로 살리기 → 글맛 안 맞으면 놓아준다
10. 공부 설명이 적용 설명보다 길면 → 비중 역전

## 톤 가이드 (공통)

- 자기비하("오만하게", "부끄럽게도") 수식은 기본적으로 넣지 않는다. 필요하면 사용자가 요청하게 한다.
- "~했다는 사실을 깨달았다"류 각성 서술은 사용자 표현이 실제로 그럴 때만 쓴다.
- 리뷰어 인용은 원문 그대로 블록 인용 `>` 로 넣는다.
- 코드 블록은 핵심만 보여준다. 전체 클래스 덤프 금지.

## 운영 팁 (토큰 효율)

회고 한 편을 끝까지 끌고 가면 cache_read 누적·재료 재투입·Phase 4 라인 단위 수정으로 토큰이 크게 부푼다. 다음 두 가지만 지켜도 비용이 눈에 띄게 줄어든다.

### 1. Phase 사이에 세션을 끊는다 (또는 `/clear`)

**Phase 2 → 3 경계는 권장이 아니라 강제다** (Phase 2 섹션의 강제 멈춤 룰). **Phase 3 → 4 (퇴고)** 경계도 세션을 끊는다. 앞 단계의 cache_read가 매 턴 누적되는 흐름이 차단되고, compact로 스킬 룰이 쓸려나간 채 작성하는 사고도 막는다.

세션 끊기는 두 방식 모두 동일한 효과를 낸다:
- 새 세션을 여는 것
- 기존 세션에서 `/clear`로 conversation history를 비우는 것 (토큰 비용 관점에서는 새 세션과 사실상 동일)

복귀 프로토콜이 이미 4파일(`회고-작업상태.md`, `회고-구조.md`, `회고-재료.md`, `회고-초안.md`)을 다 읽도록 설계돼 있기 때문에, 어느 쪽을 택하든 컨텍스트는 즉시 복원된다. 대신 직전 세션의 시도·반려된 옵션·중간 사고 과정은 자연스럽게 떨어져 나간다 — 이는 손실이 아니라 **정리** 다.

권장 분할 지점:
- Phase 0~2 (자료 + 인터뷰 + 구조) → 한 세션
- Phase 3 (꼭지별 초안) → 꼭지 묶음 단위로 세션 분리 가능
- Phase 4 (퇴고) → 새 세션 / `/clear`

### 2. 자료 투입은 URL·평문 위주로

회고 작업에서 들어오는 자료의 절대다수(리뷰 코멘트, Slack 대화, PR diff, 외부 글)는 URL이거나 평문 텍스트다. 이 경우 사용자가 **PDF·이미지·HTML을 직접 첨부할 일이 거의 없다.**

- **URL이 있는 자료**: 사용자가 URL만 던지면 skill이 WebFetch로 가져와 평문으로 컨텍스트에 넣는다. 사용자가 페이지를 통째로 복붙할 필요가 없다.
- **로컬 텍스트(메모·일기·코드 발췌)**: 마크다운이나 평문으로 본문만 붙인다.

cache_creation 토큰 효율 순서:

```
URL → WebFetch    ≈  평문/마크다운 발췌    ≪  HTML 통째    ≪  PDF·이미지
```

PDF·이미지·HTML 첨부는 정말 다른 형태로 받을 길이 없을 때(스캔본, 캡처, 외부 URL 없는 로컬 파일)만 사용한다. HTML만 가용하면 본문 inner text만 떼어 붙인다.
