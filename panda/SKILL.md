# 개발 사이클 관리 스킬 (panda)

## 이 스킬의 존재 이유

기능 단위로 작업이 쪼개지는 프로젝트에서 다음 세 가지가 자주 무너진다.

1. **요청이 휘발됨** — 대화 중에 "이것도 해줘"가 쌓이는데 어디까지 했는지 추적이 안 됨
2. **검증 없이 다음으로 넘어감** — Claude가 "했다"고 보고하면 사용자가 실제 동작을 확인하지 않은 채 진행
3. **버전이 의미를 잃음** — 임의의 시점에 버전이 올라가서 "0.x가 무엇을 의미하는가"가 흐려짐

이 스킬은 **요청 → 작업 → 검증 → 체크 → 버전업**의 사이클을 강제한다.

## 핵심 사이클

```
사용자 요청
   ↓
[1] 대/소 분류 후 features.md에 추가 (사용자에게 분류 확인)
   ↓
[2] Claude 작업
   ↓
[3] 응답 끝에 반드시: "이게 잘 작동하나요?" 질문
   ↓
[4-A] 사용자가 "잘 됨/완료/OK" → checklist에서 체크 + features.md 반영 + 버전업
[4-B] 사용자가 "하지 말자/취소" → checklist에서 항목 삭제
[4-C] 사용자가 "안 됨/수정" → checklist 유지, 다시 작업 ([2]로 돌아감)
   ↓
[5] 모든 체크리스트가 비면 → 다음 사이클 대기
   ↓
[6] 버전이 1.0 도달 → 공유 단계 안내
```

## 파일 구조 컨벤션

프로젝트 루트에 다음 2개 파일을 둔다. **토큰 판다 프로젝트의 기본 경로는 `/Users/john_park/Desktop/token-panda-v2/`** 이며, panda 파일들도 이 경로 바로 아래에 둔다.

```

├── features.md      # 진행 중 + 검증 대기 + 완료 기능을 한 파일에
└── version.md       # 현재 버전 + 변경 이력
```

**이전 위치 (deprecated, 참고용만):** `/Users/john_park/Desktop/claude-desk-pet/` — 옛 features.md / version.md 가 여기에 남아 있다. 새 작업은 절대 여기에 안 적는다. 옛 자료를 회고적으로 끌어올 일이 있으면 이 경로에서 *읽기만* 한다.

다른 프로젝트에서 panda를 쓸 때만 사용자와 위치를 합의하고, 한 번 정하면 바꾸지 않는다.

### features.md 포맷

한 파일 안에 "현재 작업 상태"와 "누적 완료 기능"을 모두 둔다.

```markdown
# 기능 & 체크리스트

## 진행 중
- [ ] [대] OAuth 로그인 추가
- [ ] [소] 로그인 버튼 색상 변경

## 검증 대기
- [ ] [소] 헤더 폰트 크기 조정  ← Claude 작업 끝, 사용자 확인 대기

---

## 완료 기능

### 대기능 (Major, +0.1)
- [v0.2] 로그인 기능
- [v0.1] 초기 라우팅 구조

### 소기능 (Minor, +0.01)
- [v0.22] 푸터 링크 추가
- [v0.21] 비밀번호 길이 검증
```

작업이 사용자 확인을 받으면 "검증 대기" → "완료 기능"의 해당 분류로 옮기고 version.md 갱신.

### version.md 포맷

```markdown
# 버전

현재: v0.22

## 이력
- v0.22 (2026-05-01): 푸터 링크 추가 [소]
- v0.21 (2026-04-28): 비밀번호 길이 검증 [소]
- v0.2  (2026-04-25): 로그인 기능 [대]
```

## 동작 규칙

### 규칙 1 — 분류는 사용자가 지정한다

분류(대/소)는 Claude가 묻지 않는다. 사용자가 요청할 때 함께 알려주는 것이 기본이다:

- "이거 추가해줘 [대]" → checklist에 [대]로 등록
- "버튼 색 바꿔줘 [소]" → checklist에 [소]로 등록
- 분류 없이 요청 → 일단 [소]로 가정해서 등록하고, 작업 시작 시 한 줄로만 알림 ("[소]로 등록했습니다 — [대]면 알려주세요")

요청이 들어오면 **작업 시작 전에**:
1. features.md의 "진행 중" 섹션에 항목 추가
2. 그 다음 작업 시작

### 규칙 2 — 응답 끝에 반드시 검증 질문

Claude가 한 작업의 결과를 보고할 때 응답 마지막 줄은 **항상** 이 형태:

> **"이거 다 됐나요? 된 항목만 알려주시면 체크하고 버전 올릴게요."**

복수 항목이 진행 중이면, 항목별로 체크할 수 있도록 번호를 붙여 보여준다:

> ```
> 진행 중:
> 1. [대] OAuth 로그인
> 2. [소] 헤더 색상 변경
> 
> 다 됐나요? 된 번호 알려주세요. (예: "1번만 OK", "둘 다 OK", "1번 다시")
> ```

이 질문이 빠지면 사이클이 깨진다. 작업이 코드 변경이 아니라 단순 답변이어도, 체크리스트에 올라간 항목이면 묻는다.

### 규칙 3 — 사용자 응답에 따른 분기

| 사용자 발화 | 처리 |
|---|---|
| "잘 됨", "OK", "완료", "맞아" | checklist에서 체크 → features.md에 옮김 → version 올림 → version.md에 기록 |
| "안 됨", "이거 수정", "다르게" | checklist 유지, 재작업 |
| "하지 말자", "취소", "빼자" | checklist에서 해당 항목 삭제 (features.md에 안 들어감, 버전 변동 없음) |
| 다른 새 작업 지시 | 현재 항목은 "검증 대기"로 옮기고, 새 항목을 "진행 중"에 추가 |

### 규칙 4 — 버전 산식

- 소기능 1개 완료 → +0.01
- 대기능 1개 완료 → +0.1
- 0.99 + 0.01 → 1.00 (자릿수 올림 정상 처리)
- 1.0 도달 시 → "공유 단계입니다" 안내하고 사용자에게 다음 행동 결정권 넘김

**[소] 작업의 범위는 "신규 기능"으로 좁히지 않는다.** 사용자에게 보이는 변화를 만드는 모든 작업이 +0.01 카운트 대상.

| 유형 | 예시 | 카운트? |
|---|---|---|
| 신규 기능 | 알림 추가, 새 버튼 | ✓ +0.01 |
| 디자인 / UI 변경 | 색상·간격·아이콘 정리, 표 재배치, 헤딩 호흡 | ✓ +0.01 |
| 버그 픽스 | 잘못된 분기 수정, 표시 오류 정정 | ✓ +0.01 |
| 리팩토링 | 로직 단순화, 중복 제거 | ✓ +0.01 |
| 카피·문서 정리 | README 톤 정리, em dash 제거 등 | ✓ +0.01 |
| 빌드/스크립트 | 파일명 규칙, dmg rename 등 | ✓ +0.01 |
| 메모리·스킬 등 외부 산출물 | 카운트 안 함(이 프로젝트 코드 밖) | ✗ |
| 의도적으로 작은 묶음 정리 | 같은 의도의 여러 변경(예: README h3 9개 일괄 br 삽입)은 1개 [소] | ✓ +0.01 |

**검증·완료 시 묶음 처리:** 사용자가 한 응답에서 여러 [소] 항목을 동시에 OK 하면, 각 항목당 +0.01씩 누적해 한 번에 올리고 version.md 이력에 같은 날짜로 N줄 기록한다. 예: 5개 OK → 0.8 → 0.85, 이력 5줄.

**버전 표기 자릿수:**
- 소기능 후 minor 부분이 한 자리로 떨어지면(0.10, 0.20 …) 트레일링 0을 유지해 비교 안정성 확보 (`0.10` ≠ `0.1` 으로 적지 않는다, 표시는 `0.10`).
- semver(`X.Y.Z`)가 필요한 곳(package.json, Cargo.toml, tauri.conf.json 등)은 panda `X.Y` ↔ semver `X.Y.0` 으로 매핑한다. 예: panda `1.17` ↔ semver `1.17.0`, panda `0.81` ↔ semver `0.81.0`. **`X.0.Y`로 매핑하지 않는다** (1.0.17은 patch=17로 해석돼 사용자 혼란).

### 규칙 5 — 새 프로젝트 / 기존 프로젝트 분기

스킬이 발동하면 가장 먼저:

1. 대상 프로젝트 경로(토큰 판다면 `/Users/john_park/Desktop/token-panda-v2/`)에 features.md / version.md가 있는가?
   - **있다** → 현재 상태 읽고 사용자에게 한 줄 요약 ("현재 v0.22, 진행 중 2건, 검증 대기 1건")
   - **없다** → 사용자에게 묻기:
     > "이 프로젝트에서 처음 panda를 쓰는 거죠? 시작 버전을 알려주세요. 그리고 지금까지 한 작업이 있으면 features.md에 회고적으로 채워 넣을지 정해주세요."

### 규칙 6 — 회고 채움 (지금까지 한 것 평가)

사용자가 "지금까지 한 것도 체크리스트에 넣어줘"라고 하면:

1. 프로젝트 디렉토리의 git log / 코드 / 메모리를 훑어서 후보 기능을 뽑는다
2. 후보 목록을 사용자에게 보여주고 **사용자가 직접** 대/소 분류와 포함 여부를 결정하게 한다
3. 확정된 것만 features.md에 채우고 version.md의 "이력"을 역산해서 채운다
4. 단, 회고 채움은 한 번만 — 이후엔 정상 사이클로만 동작

체크리스트: `package.json` / `Cargo.toml` / `tauri.conf.json` 버전 + **README 배지** + GitHub 릴리스 태그(`vX.Y.0`) 는 **한 트랜잭션**으로 같이 올린다. 어느 하나만 올리면 배지 링크 404 또는 버전 불일치가 생긴다.

**🚨 "README 무단 수정 금지" 메모리와 혼동 금지:** 사용자 메모리의 `feedback_readme_no_unsolicited_edits` 룰은 *문서 내용/구조/배지 디자인의 임의 변경*을 막는 룰이지, **release-time 버전 동기화** 와는 별개다. release 사이클(규칙 10)이 돌면 배지의 `vX.Y.Z` 텍스트와 다운로드 파일명은 **무조건 갱신** — 사용자의 "배포해줘" 한마디에 이미 동기화 의사 표시가 포함돼 있다. "건너뛰고 한 줄 보고" 같은 회피 처리 금지.

**Why:** 사용자가 2026-05-19 v1.74 배포 후 명시 정정 ("릴리즈는 됐는데 다운로드는 왜 자꾸 빼는거야?"). v1.72 에 이어 두 번째 같은 회귀라 SKILL.md 에 못박는다. README 무단 수정 금지 룰이 release 동기화에까지 잘못 확대되면 사용자 입장에선 다운로드 배지 링크가 404 가 되거나 옛 버전을 받게 된다 — 가장 비싼 회귀.

### 규칙 8 — 검증 질문 직전에 프로세스 강제 재시작

작업이 **코드 변경을 동반**했다면, "이거 다 됐나요?" 질문을 띄우기 **직전**에 무조건 재시작한다. 프로젝트 종류에 따라 두 케이스로 갈린다.

**Why (양쪽 공통):** hot-reload가 못 잡는 경계가 흔하다. Rust/네이티브 코드, 빌드 시점 const(`include_bytes!`, `env!`), 네이티브 메뉴/트레이/윈도우 빌더, webview URL 분기 등은 옛 프로세스가 옛 코드를 그대로 들고 돈다. 사용자가 "안 되는데?"라고 보고하는데 코드는 멀쩡한 상황이 가장 비싼 비용. 그래서 hot-reload 가능 여부를 추측하지 말고 **항상 kill → re-run**.

**예외 (양쪽 공통):** 코드 변경이 전혀 없는 작업(체크리스트만 정리, 빌드 산출물에 영향 없는 문서 수정)에는 적용 안 함. 애매하면 재시작이 안전.

#### 8-A. 일반 케이스 — dev 서버 재시작

데몬·서버 형태로만 돌고, 사용자가 별도 .app/.exe로 일상 사용하지 않는 프로젝트(웹 프론트, API 서버 등).

1. 해당 프로젝트의 dev/run 프로세스를 찾아 kill (떠있을 때만)
2. clean 상태에서 다시 띄움 (background)
3. 띄운 직후 검증 질문

- **Vite/Next 프론트**: `pkill -f "vite\|next dev"` → `npm run dev`
- **Spring Boot**: `pkill -f "spring-boot:run\|gradle bootRun"` → `./gradlew bootRun`
- 명령은 `package.json scripts` / `Makefile` / README에서 확인. 모르면 사용자에게 한 줄로 묻고 답을 메모리에 저장(같은 프로젝트에서 다시 묻지 않게).

#### 8-B. 데스크톱 배포 앱 케이스 — 일상 앱 통째로 갈아끼움 (Tauri/Electron 등)

토큰 판다(`token-panda-v2`, 경로 `/Users/john_park/Desktop/token-panda-v2/`)처럼 사용자가 **평소 메뉴바·도크에서 쓰는 .app/.dmg가 따로 설치**돼 있고, 그 설치본 자체를 검증해야 사용자가 안심하는 프로젝트. dev 모드를 옆에 띄우면 사용자가 평소 쓰는 메뉴바 앱과 헷갈리고, 두 개가 동시에 떠 있어 어느 게 새 코드인지 분간이 안 됨.

해야 할 일:

1. **빌드 시작 *직전* 풀 정리 (반드시 빌드 명령 호출 전)**:
   1. **bundle id로 정중하게 quit**: `osascript -e 'tell application id "<bundle.id>" to quit' 2>/dev/null` (예: `com.tnew.clauddeskpet`)
   2. **잔존 PID 강제 kill** — 1초 대기 후, ps에서 `MacOS/app` (또는 해당 binary 이름) 줄을 모두 잡아 `kill -9`. 시스템 path(`/System/`)는 제외.
   3. **옛 productName 잔재 .app 정리 + LaunchServices unregister** — productName이 한 번이라도 바뀐 적 있으면 `/Applications`에 옛 이름 .app이 남아있어 같은 bundle id의 두 번째 인스턴스로 spawn된다. 발견되면 `mv`로 `/tmp` 백업하고, **그 옛 .app과 빌드 산출물 .app(`src-tauri/target/release/bundle/macos/<옛 이름>.app`) 모두 LaunchServices DB에서 unregister**: `/System/Library/Frameworks/CoreServices.framework/Versions/A/Frameworks/LaunchServices.framework/Versions/A/Support/lsregister -u "<path>" 2>/dev/null`. unregister 안 하면 LSDB가 같은 bundle id의 옛 .app을 계속 기억해 빌드 후 cp 시점에 두 번째 인스턴스로 자동 spawn한다.
   4. **dmg 마운트 흔적 detach** — `mount | grep "<앱>"`로 `/Volumes/...`에 마운트된 dmg가 있으면 `hdiutil detach -force`.
   5. **사후 확인** — `sleep 1` 후 잔존 프로세스/마운트/잔재 .app 다시 검사. 0이 아니면 한 번 더 시도.
2. **release 빌드** — `npm run tauri:build` (또는 `npm run tauri build`). 첫 빌드 5~10분, 캐시 있을 때 1~2분. 빌드 완료까지 사용자에게 진행 알림.
3. **빌드 *후* 한 번 더 정리** — 빌드는 길다. 그 사이 LaunchServices가 빌드 산출물 .app을 인덱싱하다가 새 인스턴스를 spawn할 수 있으므로 1.1~1.2(quit + kill)만 한 번 더 돌린다.
4. **.app 교체** — 새로 빌드된 `src-tauri/target/release/bundle/macos/<앱>.app`을 `/Applications/<앱>.app`에 `mv` 백업 후 `cp -R`로 덮어쓰기.
5. **quarantine 속성 클리어** — `xattr -cr "/Applications/<앱>.app"`.
6. **단일 인스턴스 실행** — `open "/Applications/<앱>.app"`. 직후 `ps`로 binary 1개만 떠있는지 재확인. **2개 이상이면 빌드 산출물 .app도 추가로 lsregister -u로 unregister하고 잔존 PID kill 후 다시 open**.
7. **검증 질문** — "기존 일상 앱 갈아끼웠어요. 메뉴바 새 판다로 확인해주세요" 식으로 한 줄 알리고, 그 다음 평소의 진행 중 항목 검증 질문.

**Why 단계가 이렇게 많은가:** "트레이가 두 개 떠"는 가장 흔한 회귀 신호. 원인은 거의 항상 넷 중 하나 — (a) macOS LaunchServices DB가 같은 bundle id의 옛 .app을 캐싱·spawn (b) 사용자가 빌드 산출물 dmg를 더블클릭해 `/Volumes`에 마운트된 자국 (c) productName 이름이 바뀌어 `/Applications`에 옛 폴더가 잔재로 남음 (d) 빌드 산출물 폴더의 .app이 LSDB에 인덱싱돼 .app 교체 시 두 인스턴스로 spawn됨. `pkill` 한 줄은 (a)~(d) 어느 것도 못 잡는다. 특히 (a)(d)는 lsregister -u로 LSDB에서 명시적으로 빼야 하고, **빌드 직전에 한 번 + 빌드 직후에 한 번** 양쪽으로 잡아야 빌드 도중 spawn된 인스턴스도 차단된다.

**판단 기준 (8-A vs 8-B):**

| 신호 | 케이스 |
|---|---|
| 프로젝트가 Tauri/Electron, 빌드 산출물이 .dmg/.app/.exe | 8-B |
| GitHub Releases에 dmg가 올라가 있고 사용자가 /Applications에 깔아 씀 | 8-B |
| 그 외 (웹 프론트, 백엔드 서버, CLI 등) | 8-A |

**token-panda-v2는 항상 8-B**로 처리. (메모리에 박힌 토큰 판다 릴리스 동기화 룰과 같은 맥락)

**자동 적용의 조건:** 8-B는 매번 release 빌드 + 일상 앱 교체라 비용이 크다. 그래도 panda가 활성화된 데스크톱 배포 앱 프로젝트에서는 **검증 단계마다 무조건** 수행한다. 사용자가 "dev로 띄워" 라고 명시하면 그때만 8-A로 내려간다.

**🚨 절대 빠뜨리지 마라 (자주 빼먹는 케이스들):**

panda 워크플로우의 검증 단계에 들어갈 때, 아래 셋 다 항상 8-B 강행. 어느 하나라도 "이 변경은 사용자 가시 동작에 영향 없으니 빌드 의미 없다"고 판단해서 skip 하지 말 것. 사용자 입장에선 빌드+실행 시퀀스 자체가 *룰의 일관성*이라 한 번이라도 빠지면 신뢰가 깨진다.

- **production 코드 변경** — 명백히 8-B
- **테스트만 추가/수정한 변경** — production 동작 0 이어도 8-B 강행. 이유: (a) 새 테스트가 release 빌드에서도 동일하게 통과하는지 확인하는 의미 (b) 사용자 입장에서 워크플로우 일관성 (검증 단계 = 항상 빌드+실행) (c) 다음 사이클에 들어갈 때 production .app 도 최신 commit 상태여야 회귀 비교 baseline 이 명확
- **체크리스트·docs 추가만 한 변경** — 단, *코드를 한 줄도 안 건드린* 완전 메타 작업이면 8-B 면제. 애매하면 빌드.

사용자가 "빌드하고 실행해줘"라고 명시적으로 말한 적이 한 번이라도 있으면, 그 프로젝트에서는 "테스트만 추가"여도 8-B 자동 적용. 사용자가 명시한 룰을 메모리에 저장하지 않아도 SKILL.md 의 이 항목이 기본값.

**Why:** 2026-05-18에 테스트 +91건 추가하는 [대] 작업 후 검증 질문만 띄우고 빌드를 skip 했더니, 사용자가 "빌드하고 실행해줘 기능 개발할때마다 skills에 넣어줘 이 내용"으로 즉시 정정. *production 동작 변경 없음* 이라는 Claude의 판단이 panda 워크플로우의 일관성을 깨는 비용보다 작지 않다. "기능 개발할 때마다"의 정의에는 테스트 추가도 포함된다.

**🚨 사용자에게 설치를 떠넘기지 않는다 (강행 룰):**

빌드가 끝나면 절대로:
- "dmg가 `bundle/dmg/...`에 생성됐어요, 더블클릭해서 깔아주세요"
- "이 파일을 `/Applications`로 끌어다 놓으세요"
- "`open <dmg>` 명령으로 띄워보세요"
같은 말로 끝내지 않는다. 사용자의 일은 **메뉴바의 결과만 확인**하는 것까지. 설치 절차 자체는 100% Claude의 일.

Claude가 빌드 직후 무조건 수행하는 시퀀스:
1. 기존 인스턴스 정리 (위 1.1~1.2: bundle id quit + 잔존 PID kill)
2. LSDB unregister (위 1.3)
3. `/Applications/<productName>.app` rm -rf 후 빌드 산출물 .app을 `cp -R`로 깔기
4. `xattr -cr` quarantine 제거
5. **`nohup "/Applications/<productName>.app/Contents/MacOS/<binary>" > /tmp/panda-launch.log 2>&1 &` 로 binary 직접 실행** — `open` 명령은 LSDB 캐시 이슈로 silently 무시될 수 있으므로 신뢰하지 않는다. binary 직접 실행이 가장 확실.
6. 3초 대기 후 `pgrep`로 PID 확인. 0개면 fallback으로 한 번 더 시도

사용자에게 보내는 마지막 메시지는 **"메뉴바에 새 버전 떠있는지 확인 부탁"** 한 줄이면 충분하다. dmg 경로·설치 명령·Gatekeeper 우회 절차는 보고할 필요 없다 — 이 모든 게 자동으로 처리되었거나 Claude가 처리할 일이다.

**Why:** 사용자가 2026-05-16에 명시 ("내가 직접 설치하지 않을 수 있도록 / 기존 PID 찾아서 삭제 → .dmg bash로 설치하고 실행"). 솔로 데스크톱 앱 사이클에서 *빌드 후 설치 절차 안내*는 사용자가 매번 같은 행동을 반복해야 해서 가장 짜증나는 마찰점. 시퀀스가 결정론적(같은 bash 명령으로 항상 같은 결과)이라 Claude가 직접 수행하는 것이 가장 안정적.

**자동 업데이트와의 분리:** updater.rs가 호출되는 흐름(사용자가 트레이 "🆕 새 버전 설치" 클릭)에서는 *그 안에서* dmg 마운트 → cp 흐름이 일어난다. 그건 *사용자 동의 후 자동 업데이트* 경로. 빌드 직후 8-B는 *Claude가 검증을 위해 깔아 보는* 경로. 둘은 서로 다른 흐름이지만 사용자에게 dmg를 손으로 깔게 시키지 않는다는 원칙은 동일하다.

### 규칙 9 — 회귀 방지 게이트 (테스트 + 스모크 체크리스트)

새 기능을 넣을 때마다 기존 기능이 깨지는 걸 막기 위한 2단계 게이트. **코드 변경을 동반한** 작업의 검증 흐름에 박는다.

**9-A. 자동 게이트 — `npm test`** (또는 프로젝트의 동등 명령)

- 규칙 8(프로세스 재시작) 완료 후, "이거 다 됐나요?" 검증 질문을 띄우기 **직전**에 `npm test`(또는 `cargo test` 같은 프로젝트별 테스트 명령)를 무조건 실행한다.
- **실패 시 검증 질문 띄우지 않는다.** 실패한 테스트 이름과 출력을 사용자에게 그대로 보고하고, 회귀인지/이번 변경의 의도된 동작 변경인지 사용자에게 묻는다.
  - 회귀 → 코드를 고친다 (재작업, 사이클 [4-C])
  - 의도된 동작 변경으로 보이는 경우 → **9-E를 따른다 (기본은 코드를 다시 설계해 기존 테스트가 통과하게 만든다)**
- 테스트 명령은 `package.json scripts` / README / 프로젝트 설정에서 확인. 모르면 사용자에게 한 줄 묻고 메모리에 저장.

**9-B. 수동 게이트 — `regression-checklist.md` 스모크 시나리오**

- 프로젝트 루트의 `regression-checklist.md`에 사용자 손으로 훑을 카테고리별 시나리오가 있다. 매번 전체를 다 돌리지 않는다.
- 8-B(데스크톱 앱 갈아끼움) 완료 후, 검증 질문에 **이번 변경과 관련된 카테고리 번호만** 짚어준다.
  - 예: 트레이 코드 건드린 변경이면 "체크리스트 1번(메뉴바·트레이) + 5번(에너지 상태 전환의 트레이 % 동기화) 빠르게 훑어주세요"
  - 어느 카테고리가 영향 범위인지 판단은 Claude가 한다. 변경 파일과 카테고리 매핑을 머릿속에서 굴려라.
- 사용자가 회귀를 발견하면 → 그 항목을 새 [소]로 features.md "진행 중"에 등록.

**9-C. 체크리스트 성장**

- 새 [대] 또는 새 카테고리 [소]가 들어가 사용자 확인까지 받았다면, 같은 [소] 안에서 **`regression-checklist.md`에 해당 시나리오도 같이 추가**한다. 체크리스트는 코드와 함께 성장하는 자산.
- 이 추가는 별도 [소]로 분리하지 않고 본 작업에 묶는다 (분리하면 체크리스트가 영구히 한 버전 뒤처짐).

**9-D. 테스트 성장 — 새 기능엔 단위 테스트 동반 (강제)**

새 [소]/[대]가 **테스트 가능한 영역**을 건드리면 같은 작업 안에 단위 테스트도 같이 작성해 커밋한다. 별도 [소]로 분리하지 않는다 (분리하면 영구히 뒤처짐).

**판단 기준 — 테스트 가능 vs 스모크 대체:**

| 영역 | 처리 |
|---|---|
| 순수 함수, 결정론적 로직 (petLogic.ts, 포맷터, 비교 함수, derived state 계산 등) | **vitest 단위 테스트 강제** |
| 상태 머신 / 분기 트리 (예: 새 PetState 임계 추가, 마이그레이션 로직 분기) | **vitest 단위 테스트 강제** |
| Store 마이그레이션·자료 변환 (예: 새 legacy → new 키 변환) | **vitest 단위 테스트 강제** (Tauri store는 모킹 또는 pure 변환 함수로 분리) |
| React 컴포넌트의 시각적 동작 | 체크리스트 9-B로 대체 가능. 단, 컴포넌트 안의 *순수 헬퍼*를 별 파일로 빼낼 수 있으면 빼내서 테스트한다 |
| Tauri Rust 커맨드 / 트레이 / 윈도우 빌더 / IPC | 체크리스트 9-B 대체. 단, Rust 쪽 pure 로직(버전 비교·URL 파싱·자료 매핑 등)은 `cargo test`로 단위 테스트 강제 |
| 외부 API 호출 (GitHub releases 등) | **fetch 함수와 응답 파싱을 분리하여 파싱부만 단위 테스트**. fetch 자체는 체크리스트 |

**실제 적용 흐름:**
1. 새 [소] 등록 후, 코드 변경을 시작하기 전에 *어디에 테스트가 들어갈지* 한 줄로 사용자에게 보고. 예: "이번 변경은 `compareVersions()` 새 함수가 핵심이라 `versionCompare.test.ts` 같이 만들겠음. UI 표시는 스모크 체크리스트 1번에 항목 추가."
2. 구현 + 테스트를 같은 응답 안에서 끝낸다
3. 9-A의 `npm test` 게이트에서 신규 테스트도 같이 통과해야 검증 질문 띄움

**예외:** 변경이 *완전히* UI/IPC만 건드리고 순수 로직 0 줄이라면 스모크만으로 OK. 이 판단을 사용자에게 한 줄 알리고 진행 (사용자가 "그래도 어떻게든 테스트 만들어"라고 요구하면 그때 추출 리팩토링부터).

**Why:** 9-A(기존 테스트 통과)만으로는 새 기능의 회귀를 못 잡는다. *추가되는 기능 자체*에 안전망을 같이 깔아야 다음 변경이 이걸 깨면 즉시 발견된다. 사용자가 2026-05-15에 명시 ("기능 추가될 때마다 기존 게 깨지는 게 싫다" + "모든 새로운 기능 추가시 테스트 진행").

**9-E. 기존 테스트는 절대 건드리지 않는다 (immutable)**

기존 테스트 파일의 케이스(`it(...)` / `#[test]` 블록)의 **expected 값, 입력값, 의도, describe 그룹 구조**는 어떤 경우에도 **수정·삭제하지 않는다**. 새 케이스를 같은 파일/모듈에 *추가*하는 것만 허용.

**예시:**
- ✅ `petLogic.test.ts`에 `it("새 케이스", ...)` 한 블록 추가
- ✅ 새 `*.test.ts` 파일 신규 생성
- ❌ 기존 `expect(d.petState).toBe("sleepy")`를 `"tired"`로 변경
- ❌ 기존 `describe`를 통째로 삭제
- ❌ 기존 fixture JSON을 다른 값으로 갱신
- ❌ "이름만 더 명확하게 바꾸겠음" 같은 리팩토링도 X (이름 바꾸는 것도 의도 변경으로 간주)

**기존 테스트가 새 변경과 충돌해 깨지면:**

먼저 **코드 쪽을 다시 설계해 기존 테스트가 그대로 통과하게 만든다.** 임계치 추가, 새 분기 삽입, 함수 시그니처 확장 등으로 기존 동작을 보존하면서 신기능을 얹는다.

그렇게 해도 불가능한 *진짜 동작 변경*인 경우에만:
1. 사용자에게 한 줄로 명시적으로 **"이 변경이 기존 테스트 N건(이름 나열)의 expected를 깨뜨릴 예정입니다. 같이 업데이트해도 되나요?"** 라고 묻는다
2. 사용자가 명시적으로 OK한 경우에만 기존 테스트를 수정한다
3. 묻지 않고 기존 테스트를 수정·삭제하는 것은 룰 위반

**Why:** 기존 테스트는 과거 결정의 명세이자, *코드 변경이 회귀를 일으켰는지*를 검출하는 안전망. Claude가 "이건 의도된 변경이니까"라고 판단해 기존 테스트를 같이 고쳐버리면, 실제로는 회귀였는데 안전망까지 같이 무너지는 케이스가 잡히지 않는다. 사용자가 2026-05-15에 명시 ("기존에 있던 테스트는 절대 건들이지 마").

**9-F. 업데이트 일지(changelog) 성장 — 사용자 가시 변화는 같이 기록 (강제)**

사용자 화면·동작에 드러나는 변화를 만든 [소]/[대]가 검증·완료되면, **같은 작업 안에서** `src/changelog.ts` 의 `CHANGELOG` 배열 맨 앞에 항목을 추가한다. 별도 [소]로 분리하지 않는다 (분리하면 일지가 영구히 뒤처진다 — 9-C 체크리스트·9-D 테스트 성장과 같은 결: 자산은 코드와 함께 자란다).

**넣는 것 (앱 화면에 드러나는 변화):** 새 기능, 새 캐릭터/스킨, UI·표시 변경, 눈에 띄는 버그 픽스, 새 provider/계정 지원 등.

**빼는 것 (사용자 화면에 안 드러나는 내부 변경):** 리팩토링, 테스트 추가, 빌드/CI, 내부 식별자·영문 병기, 텔레메트리 백엔드, repo 리네임, README·문서만 고친 patch. (version.md 에는 남기되 changelog 에는 안 넣는다.)

**표기 규약:** `version` 은 semver `X.Y.0` (panda `X.Y` ↔ `X.Y.0`). 최신이 맨 앞 = 내림차순 (`changelog.ts` 무결성 검사가 강제 — 어기면 `npm test` 실패). `date` 는 커밋 날짜. `title` 은 접힘 상태 한 줄, `body` 는 펼침 마크다운으로 **친근체("~어요/~습니다") + 짧은 불릿**, 기술 용어 대신 사용자가 체감하는 표현으로. (version.md = 내부 기술 기록 / changelog.ts = 사용자용 일지, 둘은 분리.)

**외부 기여 PR:** PR diff 에는 changelog 를 넣지 않고, 메인테이너(사용자)가 머지 후 버전 bump 와 함께 일지를 반영한다 (예: v2.32 PR #1 — 기능 코드는 머지 커밋, 버전·changelog 는 별도 커밋).

**검증:** changelog.ts 는 앱 번들 코드라 9-A(`npm test`)에 포함된다 — 내림차순·필수필드 무결성 테스트가 같이 돈다.

**Why:** version.md(내부 기술 기록)와 changelog.ts(사용자용 일지)가 분리돼 있어, 기능을 넣고 일지를 빼먹으면 사용자가 트레이 "업데이트 일지"에서 새 기능을 못 본다. v2.25 이후 v2.26~v2.31 이 백필 안 돼 v2.32 시점에 한꺼번에 메운 전례가 있다. 사용자가 2026-06-17 에 명시("기능 만들어질 때마다 업데이트 일지 최신화하는거 md 파일에 넣어줘").

**비활성화 조건:**
- 코드 변경이 없는 작업(문서·체크리스트만 정리 등)에는 적용 안 함 — 규칙 8 예외와 동일
- 사용자가 명시적으로 "이번엔 테스트/체크리스트 빼고 진행"이라고 한 경우 그 응답에서만 생략

**Why:** 솔로 데스크톱 앱은 회귀 발견이 늦으면 사용자 본인이 평소 메뉴바에서 쓰다가 어느 날 깨진 상태로 발견하게 된다. 자동(단위 테스트) + 수동(스모크) 2단 게이트로 검증 단계에서 잡는 비용이 가장 싸다. 사용자가 2026-05-15에 "기능 추가될 때마다 기존 게 깨지는 게 싫다"고 명시.

### 규칙 10 — "배포해줘" 명시 시 release 까지 완주

사용자가 "배포해줘" / "release 해줘" / "GitHub 에 올려" / "Latest 로 풀어줘" 같은 표현을 쓰면, *그 한 단어가 의미하는 모든 단계를 끝까지 수행*한다. 빌드까지만 하고 멈추거나, push 까지만 하고 멈추지 말 것. 사용자 시각에 "배포 = 다른 사람도 받아갈 수 있는 상태" 까지가 한 단위.

**전제**: [대] 마디가 사용자 OK 까지 받은 상태 + 8-B 빌드/설치 사이클이 완료된 시점. 즉 *이미* features.md / version.md / regression-checklist 가 최종 정리되어 있고, 메뉴바에 새 버전이 떠 있는 상태.

**수행 시퀀스 (한 단계도 건너뛰지 말 것)**:

1. **git 상태 확인** — `git status --short` + `git log --oneline -5`. 변경 파일과 기존 commit 톤 파악.

2. **모든 변경 stage 후 commit** — 한 [대] = 한 commit 이 이상. 같은 사이클에 여러 [대] 가 묶여 있으면 메시지 body 로 분리 표기. 메시지 톤은 *그 프로젝트의 기존 commit 패턴* 을 그대로 따른다(예: token-panda 는 `feat: <한국어 한 줄> v<X.Y>` 한 줄, Co-Authored-By 없음). 사용자별 / 프로젝트별 룰이 따로 있으면 그 룰 우선.

3. **태그 생성 + push** — `git tag vX.Y.0` → `git push origin main` → `git push origin vX.Y.0`. panda `X.Y` ↔ semver `X.Y.0` 매핑 그대로.

4. **CI release workflow 대기** — 프로젝트에 `.github/workflows/release.yml` 같은 자동 release 셋업이 있으면 태그 push 가 trigger 한다. `gh run list --limit 1` 로 in_progress 확인 → `gh run watch <id> --exit-status` 로 완료 대기. 보통 macOS arm64 + Windows x64 매트릭스라 10~15분.

5. **CI 가 만든 release 검증** — `gh release view vX.Y.0` 로 상태 확인. 흔히 발생하는 두 문제:
   - **태그 미연결 (`untagged-xxx`)**: tauri-action 이 draft 만들 때 가끔 태그 충돌 회피 모드로 떨어짐. `gh release edit untagged-xxx --tag vX.Y.0` 로 연결.
   - **자산 이름 깨짐**: productName 에 한글/공백이 있으면 GitHub asset 업로드 시 prefix 가 잘려 `_X.Y.0_arm64.dmg` 같은 형태로 올라간다. workflow 의 rename step 이 *빌드 후 upload 전* 이 아닌 *upload 후* 에 돌면 release 의 asset 이름은 안 바뀜. `gh api -X PATCH /repos/JohnPrk/token-guardians/releases/assets/<asset_id> -f name="token-panda_X.Y.0_arm64.dmg"` 로 각 자산 PATCH (자산 ID 는 `gh release view --json assets` 에서).

6. **draft → published 전환** — `gh release edit vX.Y.0 --draft=false --title "vX.Y.0" --notes "..."`. 노트는 짧게(한 줄). 자세한 변경은 version.md 참고 안내.

7. **Latest 등록 확인** — `gh release list --limit 3` 의 첫 줄에 `Latest` 라벨 + 새 버전. 다른 prerelease 등이 위에 있으면 처리.

8. **사용자에게 release URL 명확히 안내** — `https://github.com/<owner>/<repo>/releases/tag/vX.Y.0` 형태. 자산 이름·크기·다운로드 경로도 한 줄로 정리. 사용자가 GitHub UI 캐시로 못 봤다고 다시 묻는 경우를 막기 위해 *명확히* 표기.

9. **(선택) 자동 업데이트 흐름 환기** — 프로젝트에 자체 updater(예: token-panda updater.rs)가 있으면 "기존 사용자는 트레이 헤더의 🆕 마커로 받음" 한 줄 보고.

**README 배지 (예외 없음)**: 규칙 7 그대로 — **release 사이클이 돌면 README 의 다운로드 배지 2개(.dmg + Windows)는 무조건 갱신**. 사용자 메모리의 "README 무단 수정 금지" 룰은 *임의 콘텐츠/구조 변경* 에 대한 것이지 release-time 버전 동기화 와는 별개다. "건너뛰고 한 줄 보고" 같은 회피 처리 금지. 같은 commit 에 묶을 수 없으면(예: 메인 release commit 이 이미 push 된 후) **별도 `chore: README 다운로드 배지 vX.Y.Z → vA.B.C 동기화` commit 으로 즉시 push**. release URL 안내 전에 끝낸다.

**자주 빼먹는 케이스 (강조)**: Download Windows 배지도 동일하게 vX.Y.0 으로 갱신. .dmg 만 바꾸고 Windows 를 빼먹는 패턴이 v1.72 / v1.74 두 차례 반복. 두 배지 한 줄씩 같이 처리.

**Why**: 사용자가 2026-05-18 에 명시("내가 배포 해달라고 하면 release까지 할 수 있도록 skills 넣어줘"). v1.70.0 작업 후 build + commit + tag push 까지만 하고 *draft release 의 asset rename + publish* 단계를 사용자에게 떠넘긴 게 정정 원인. "배포" 의 정의에 publish + asset 이름 정상화 + Latest 등록이 다 포함된다는 것을 SKILL.md 에 박는다.

**과용 금지**: 사용자가 "빌드해줘" / "설치해줘" / "검증해줘" 같이 *배포 의도가 명시 안 된* 표현을 쓸 때는 이 시퀀스 진입하지 않는다(8-B 사이클만). "배포" / "release" / "공유" / "올려" 류 명시적 표현일 때만.

## 안 하는 것

- 사용자 확인 없이 임의로 버전을 올리지 않는다
- 사용자가 "잘 됨"이라고 하지 않은 항목을 features.md에 옮기지 않는다
- 검증 질문을 생략하지 않는다 (응답이 길어도 마지막 줄은 무조건 그 질문)
- 1.0 도달 시 자동으로 다음 단계로 넘어가지 않는다 — 사용자에게 결정권을 넘긴다
- 학습 노트, 회고 글 같은 비-기능성 작업에는 쓰지 않는다 (별도 스킬이 있음)

## 발동 / 비발동

**발동**:
- 사용자가 features.md + version.md가 있는 디렉토리에서 작업 지시를 함
