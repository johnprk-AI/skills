# 기술 학습 노트 작성 스킬

## 이 스킬의 존재 이유

`mission-retrospective` 스킬은 "내가 한 일과 그때의 감정"을 살리는 데 최적화돼 있다. 그 톤을 기술 정리에 끌고 들어오면 두 가지가 망가진다.

1. **알아낸 것보다 "알아낸 척"이 더 많아진다.** 감정 아크를 맞추려고 사용자가 실제로는 거치지 않은 시도·검증이 끼어든다.
2. **수치가 분위기에 휩쓸린다.** 기술 글에서 "충격이었다", "엄청 신기했다" 류 수식이 들어가면 그 자리의 숫자가 함께 부풀려진다.

이 스킬은 반대 방향으로 잡혀 있다.

- 인터뷰 X. 감정 아크 X. 4파일 분리 X.
- "내가 무엇을 보고 있었나 → 어디서 의문이 생겼나 → 공식 문서/실측치로 어떻게 풀었나 → 무엇이 남았나"의 단순한 동선.
- **수치는 실측치. 공식 문서는 짧게.**

## 발동 조건

다음 중 두 개 이상 해당하면 발동:

- 사용자가 어떤 기술 주제(prompt caching, JDBC 내부, OAuth 흐름, GC 동작 등)를 *최근 작업 중에 마주쳤다고* 말함
- 사용자에게 그 주제와 관련된 **공식 문서 URL**이 있음
- 사용자에게 본인 작업의 **실측 자료**(로그 파일·메트릭·캡처·스크립트 출력 등)가 있음
- 사용자가 "정리해서 공유하는 느낌", "담담하게", "질답 말고 정리"라는 톤을 명시함

회고 스킬과 동시 발동 가능 조건이면 **사용자가 명시한 쪽**을 따른다. 둘 다 명시하지 않았는데 자료가 PR·리뷰 코멘트 위주면 회고 쪽으로 양보한다.

## 4단계 프로토콜

```
Phase 0 │ 자료 + 시작점       — 공식 문서 / 실측 자료 / 한 줄 발단
Phase 1 │ 학습 동선 합의      — 시작점 → 의문 → 풀이 → 결론
Phase 2 │ 본문 작성           — 한 호흡, 옵션 X
Phase 3 │ 수치·문서 검증      — 표 단위까지 대조
Phase 4 │ 발행 (옵션)         — 사용자가 "배포해줘"라고 명시할 때만
```

회고 스킬과 달리 Phase 1 합의 후 **곧바로 본문 작성**으로 간다. Phase 4(발행)는 사용자가 명시적으로 요청할 때만 발동.

## Phase 0 — 자료 + 시작점

다음을 한 번에 받는다.

1. **공식 문서 URL** (1~3개)
2. **실측 자료** — jsonl 로그, 콘솔 출력, 그래프 캡처 등. *이 스킬의 본체는 실측치이지 공식 문서가 아니다.*
3. **한 줄 발단** — 왜 이걸 알아보게 됐는지

발단이 모호하면 한 번 더 묻는다. "어떤 작업을 하다가 이 주제를 보게 됐어요?" 발단이 없으면 글의 진입점이 일반론으로 흐른다.

자료 형태별 효율 순서(cache_creation 단가 기준):

```
URL → WebFetch    ≈  평문/마크다운 발췌    ≪  HTML 통째    ≪  PDF·이미지
```

PDF·이미지는 다른 형태가 없을 때만.

## Phase 1 — 학습 동선 합의

다음 4점을 한 줄씩 박아 사용자 OK를 받는다.

1. **시작점** — 어떤 작업을 하던 중이었나
2. **의문** — 어디서 "어, 이게 뭐지?"가 생겼나
3. **풀이 경로** — 공식 문서 어느 부분 + 실측 자료 어느 부분으로 풀었나
4. **결론** — 글을 다 읽은 사람이 가져갈 한 문장

회고 스킬의 "메타 테마 한 문장"에 대응하는 자리. 이 한 문장이 흔들리면 본문도 흔들린다.

## Phase 2 — 본문 작성 (한 호흡)

### 구조 가이드

- **서두**: 시작점을 한 단락으로. "어떤 작업을 하다가 무엇을 마주쳤다."
- **발견**: 의문이 생긴 자리. 수치·로그·캡처를 직접 보여준다.
- **원리**: 공식 문서 인용으로 풀이.
- **실측 매칭**: 공식 문서가 말하는 게 내 실측치에 어떻게 맞아떨어지는지. *이 자리가 글의 차별점.*
- **정리**: 글을 다 읽은 사람에게 남길 통찰들. 단편 3~5개.
- **다음 룰**: 다음에 쓸 때의 자기 룰을 짧게.

### 작성 규칙

- **1인칭이지만 자랑·각성 없음.** "충격이었다", "신기했다", "엄청 놀랐다" 류 수식 금지. 사실만으로 충격을 전달한다.
- **em dash(`—`, U+2014) 금지.** 본문·제목·소제목 모두. 한국어 글에서 영어 번역체처럼 들리고 톤이 인위적으로 흐른다. 쉼표·마침표·괄호·콜론·줄바꿈 등 자연스러운 한국어 구두점으로 대체. 인용문 안의 원본 em dash는 원문 보존(의역 금지 룰 우선).
- **공식 문서 인용은 블록 인용(`>`)으로**, 가급적 한국어 문서 기준. 한 인용은 2~3줄 이내. 의역 금지.
- **수치는 실측치만.** 추정이면 "대략" "약"을 명시하고 출처를 표기.
- **표는 자연스러운 자리에서만.** 매 섹션 표·코드 블록 강제 X.
- **A/B/C/D 옵션 금지.** "더 풍성하게" 요청에는 새 자료를 요구한다 (회고 스킬 룰 그대로).
- **한 꼭지를 시작하면 끝까지 한 호흡.** 중간 확인 질문 X.
- **친절한 정리·요약 박스 남발 금지.** 본문이 이미 정돈돼 있으면 끝.

### 톤 — 회고 스킬과의 차이

| 항목 | 회고 스킬 | 이 스킬 |
|---|---|---|
| 어투 | 1인칭 + 감정 동반 | 1인칭 + 사실 위주 |
| 시제 | 과거 회상이 주 | 과거(작업) + 현재(정리) 병행 |
| 인용 | 리뷰어·페어 인용 | 공식 문서·실측 로그 인용 |
| 마무리 | 감정 착지·다음 미션 예고 | 단편 정리 + 다음 룰 |
| 서두 | 미션 도입·기대감 | 작업 맥락 한 단락 |

## Phase 3 — 수치·문서 검증

본문 작성 후 다음을 점검한다.

- [ ] 모든 수치에 **출처**가 있는가 (jsonl 어느 세션, 어느 표, 어느 명령 출력)
- [ ] 단위가 일관되는가 (토큰/byte/char/줄)
- [ ] 공식 문서 인용이 원문에서 검증 가능한가 (의역하지 않았는가)
- [ ] **추정 vs 실측**이 본문에 분명히 드러나는가
- [ ] 약어·전문 용어는 첫 등장 시 풀어 적었는가 (TTL = Time To Live, MCP = Model Context Protocol 등)
- [ ] 본인이 안 한 시도·검증을 슬그머니 끼워 넣지 않았는가
- [ ] 시간대를 표기할 때 사용자 로컬 타임존(KST 등)으로 통일했는가

회고 스킬의 "사용자 표현이 아닌 시도를 끼워 넣었나"와 같은 정신. 검증 기준이 *경험* 대신 *수치와 문서*라는 점만 다르다.

## Phase 4 — 발행 (옵션)

사용자가 "배포해줘", "푸시까지 해줘" 등으로 *명시*하면 진입. 그렇지 않으면 Phase 3에서 종료하고 파일만 남긴다.

### 사전 확인 — 블로그 publishing 메모리 룩업

활성 프로젝트의 memory 디렉토리에서 다음 이름들을 순서대로 찾는다:
- `reference_blog_publishing.md`
- `user_profile_blog.md`
- `user_profile.md`

찾으면 그 메모리의 *레포 경로 / frontmatter 스키마 / 카테고리 / 커밋 컨벤션 / 배포 방식*을 그대로 따른다. 못 찾으면 사용자에게 한 번에 묻는다 ("레포 경로 / 카테고리 / 커밋 스타일 / 배포 방식"). 묻고 받은 답은 그 자리에서 *reference 메모리로 만들어둔다* — 같은 정보를 두 번 묻지 않기 위해.

### 발행 단계

1. **em dash 0회 확인** (`grep -n $'—' <new-post>.md`). 한 건이라도 있으면 발행 중단, 본문 수정 후 재진입.
2. **빌드 통과 확인.** Next.js라면 `npm run build`, Hugo라면 `hugo`, Jekyll이라면 `jekyll build`. 빌드 산출물에 새 글의 페이지가 생성됐는지 한 번 확인.
3. **명시적 파일 add.** `git add <new-post>.md`만. `git add .`이나 `-A`는 빌드 산출물·IDE 설정 파일까지 끌고 갈 수 있어 금지.
4. **커밋.** 메모리에 박힌 컨벤션을 따른다. 시리즈여도 *글 1편 = 1커밋* 분리가 보통.
5. **푸시.** `git push origin main` (또는 default branch).
6. **CI 큐인 확인.** `gh run list --limit 1`로 배포 잡이 큐에 올라갔는지 한 줄 확인. 끝까지 기다리지는 않는다.

### 발행 직후 사용자 보고

발행한 글 목록을 마크다운 링크로 정리. 본인 자료 인용 위치(PR 번호·커밋 SHA·실측 출처)를 함께 명시해서, 사용자가 글의 출처를 한 번에 확인할 수 있게.

## 운영 팁

- 글이 짧을 수밖에 없는 자리에서 **억지로 늘리지 않는다.** 회고 스킬과 동일.
- 이 스킬로 쓴 글의 자연스러운 길이는 **150~350줄(약 6,000~14,000자) 정도.** 그 이상 부풀면 다른 주제가 끼어든 신호.
- **공식 문서 인용보다 사용자가 실제로 본 것이 무겁다.** 공식 문서는 보강, 실측치가 본체.
